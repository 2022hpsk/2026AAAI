# Part 1: SI-RNN（AAAI 2018）—— AAAI 自家经典

## 1.1 为什么这篇特别重要

SI-RNN 是 AAAI 2018 的论文——**AAAI 自家发表的多方对话经典**。这有两层意义：

1. **学术 legitimacy**：证明 speaker-grounded 多方对话研究在 AAAI 有传统
2. **审稿口味**：AAAI 审稿人会认识这篇，cite 它能贴合审稿期待

## 1.2 核心方法

SI-RNN 处理多方对话中的两个 task：

- **Addressee selection**：每条 utterance 指向谁
- **Response selection**：选择正确的下一条 response

### 1.2.1 三角色建模

SI-RNN 引入 **sender / addressee / observer** 三角色：

- **Sender**：当前 turn 的说话人
- **Addressee**：被说话的对象
- **Observer**：旁观者（也能听到，但不是被针对）

每个 utterance 会**改变所有人的角色**——比如 Alice 对 Bob 说话时：

```
turn t:
  Alice → sender
  Bob → addressee  
  Charlie, David → observer

turn t+1 (Bob 回应 Alice):
  Bob → sender
  Alice → addressee
  Charlie, David → observer

turn t+2 (Charlie 插话回应 Alice 的话题):
  Charlie → sender
  Alice → addressee (now)
  Bob, David → observer
```

### 1.2.2 Role-sensitive 更新

SI-RNN 用 **role-sensitive RNN** 更新每个 speaker 的 embedding。具体：

```python
# 每个 speaker 有一个 embedding e_s
# 每个 turn 更新 embeddings 时，根据当前角色用不同的 RNN cell

for turn in dialogue:
    sender = turn.speaker
    addressee = turn.addressee
    observers = all_others - {sender, addressee}
    
    # Sender's embedding update: 强 update (their own utterance)
    e[sender] = RNN_sender(e[sender], turn.utterance)
    
    # Addressee's embedding update: 中度 update (directed at them)
    e[addressee] = RNN_addressee(e[addressee], turn.utterance)
    
    # Observers: 弱 update (overheard)
    for obs in observers:
        e[obs] = RNN_observer(e[obs], turn.utterance)
```

这种 **role-conditional** 更新让 embedding 反映每个 speaker 的 actual experience of this conversation.

## 1.3 对你做多角色记忆的启示

### 1.3.1 role 三分法是 attention 的设计原则

你的记忆系统也可以借鉴：

- speaker 自己发言的内容 → high attention，深刻记忆
- 被对其说话的内容 → high attention（应该 update 自己的 understanding）
- 旁观到的内容 → 中等 attention（也要记，但 priority 低）

这种 role-conditional memory weighting 是非常自然的。

### 1.3.2 引用 AAAI 经典的写作策略

在你的 Related Work 里：

> "The importance of speaker-grounded modeling in multi-party dialogue has been recognized at least since SI-RNN [Ouchi & Tsuboi, AAAI 2018], which introduced role-sensitive RNN updates that distinguish sender, addressee, and observer perspectives. We extend this principle to multi-party memory management..."

这种 **回溯 AAAI 自家经典** 的写法让 AAAI 审稿人感到熟悉/被尊重——非常有效的 reviewer-friendly 写作技巧。

# Part 2: WebAgent-R1（EMNLP 2025）—— M-GRPO 工程化

## 2.1 核心贡献

WebAgent-R1 在 web agent 任务上做 RL，提出三个工程化技巧：

### 2.1.1 M-GRPO（Multi-turn GRPO）

GRPO 原本是 single-turn 的（DeepSeek-R1 用于 math）。M-GRPO 把它扩展到 multi-turn agent：

```python
def m_grpo_step(query, num_groups=8, max_turns=20):
    trajectories = []
    for i in range(num_groups):
        tau = []
        state = initial_state
        for t in range(max_turns):
            action = sample_action(policy, state)
            new_state, reward = env.step(action)
            tau.append((state, action, reward))
            state = new_state
            if is_terminal(state):
                break
        trajectories.append(tau)
    
    # 关键：在 multi-turn 下，trajectory-level reward 怎么算
    rewards = [sum(t.rewards) for t in trajectories]  # 或最终 reward
    advantages = standardize(rewards)
    
    # 每条 trajectory 的所有 turn 的所有 token 共享 trajectory advantage
    for tau, A in zip(trajectories, advantages):
        for turn in tau:
            for token in turn.action:
                update_policy(token, A)
```

这种 multi-turn 扩展和 MemSearcher 的 multi-context GRPO 是相似 idea。

### 2.1.2 动态上下文压缩

Web agent 在长 task 上 context 爆炸。WebAgent-R1 的解法：

- 每 K turn 检查 context 长度
- 超过阈值时，调用 LLM compress 历史
- 保留 recent K turn raw + 早期 compressed summary

这种 dynamic compression 让 token cost 受控。

### 2.1.3 异步 rollout

GRPO 训练需要 sample G 条 trajectory。如果同步等所有 trajectory 完成，慢的拖累快的。WebAgent-R1 用异步 rollout：

- 每条 trajectory 独立 sample
- 完成一条就 enqueue 到 training batch
- 累积到 batch size 触发 gradient update

这种异步可以提升 30-50% 训练吞吐。

## 2.2 对你做多角色 RL 训练的应用

WebAgent-R1 的三个技巧**全部直接适用**于你的多角色场景：

| 技巧                | 你的应用                                   |
|---------------------|--------------------------------------------|
| M-GRPO              | 你的 multi-turn 多方记忆 RL 训练           |
| 动态上下文压缩      | 群聊 raw history 压缩，保留 memory + recent |
| 异步 rollout        | 你的训练吞吐优化                            |

**这些是工程必备**——AAAI 论文 method 章节里应该至少 mention 这些技巧。

# Part 3: RAGEN（arXiv 2504.20073）—— "Echo Trap" 预警

## 3.1 什么是 Echo Trap

RAGEN 揭示了 multi-turn RL 训练的一个 specific failure mode："**Echo Trap**"。

### 3.1.1 现象

在 multi-turn RL 训练后，policy 可能**坍缩到重复 template**：

```
turn 1: "Let me think about this. [pattern X]"
turn 2: "Let me think about this. [pattern X]"
turn 3: "Let me think about this. [pattern X]"
... (模型在所有 turn 重复同一 pattern)
```

模型变得 **deterministic 且 boring**，失去 exploration。

### 3.1.2 为什么会发生

multi-turn RL 的 reward 是 trajectory-level 的。如果某个 pattern 在训练初期偶然 work 好，advantage 高，policy 会 over-amplify。一旦 amplification 启动：

- 模型在 turn 1 高概率输出该 pattern
- pattern 反馈到 turn 2 的 input
- turn 2 也输出 pattern（因为 input 类似）
- 形成自循环 → echo

### 3.1.3 RAGEN 的 fix

RAGEN 提出几个 mitigation：

1. **KL 正则化**：让 policy 不能偏 ref 太远，保留 diversity
2. **多样性 reward**：显式奖励 turn 之间的多样性
3. **Temperature schedule**：训练初期高 temperature exploration

## 3.2 对你的关键意义

**这是你 multi-turn RL 训练必须正面应对的 already-known pitfall**。

在论文里要明确写：

> "Multi-turn RL training is known to suffer from the 'Echo Trap' phenomenon [RAGEN, 2025], where the policy collapses to repetitive templates. We mitigate this through (1) KL regularization with coefficient κ=0.1, (2) explicit diversity reward weighted at γ=0.05, and (3) temperature scheduling during early epochs."

**不写 = 审稿人会问 "did you address Echo Trap?"——挨拒**

# Part 4: 多 turn RL 实践指南（arXiv 2510.01132）

## 4.1 核心贡献

这是一份 systematic 的 multi-turn RL **algorithm comparison**。比较 PPO/RLOO/GRPO/DAPO 在 multi-turn 场景的适配问题。

### 4.1.1 关键发现

| 算法     | Single-turn 稳定性 | Multi-turn 稳定性 | Echo Trap 风险 | 推荐场景            |
|----------|-------------------|-------------------|----------------|---------------------|
| PPO      | 高                | 中                | 低             | 经典 baseline       |
| RLOO     | 中                | 中                | 中             | 内存受限场景        |
| GRPO     | 高                | 中-高（需 mod）   | 中             | math reasoning, RL-r1 |
| DAPO     | 中                | 高                | 中-低          | long-horizon agent  |

### 4.1.2 实践建议

- **Single-turn task**：用 GRPO，稳定 + 高 sample efficiency
- **Multi-turn 但 short（< 10 turn）**：用 GRPO with M-GRPO 扩展
- **Multi-turn long（> 20 turn）**：用 DAPO 或自定义 step-wise GRPO
- **Memory-management RL**：用 step-wise GRPO（AgeMem 风格）

## 4.2 对你的应用

你做多角色 RL 训练时的算法选择：

- **短群聊（< 20 turn）**：M-GRPO（WebAgent-R1 扩展）+ multi-context GRPO（MemSearcher 扩展）
- **长群聊（> 50 turn）**：step-wise GRPO（AgeMem 风格）
- **混合**：用一个 main algorithm + 多个 modification

在你的 method 章节里 cite 这份指南：

> "Following the systematic algorithm comparison of [Multi-turn RL Practical Guide, 2025], we adopt step-wise GRPO with M-GRPO extension as our base algorithm, which provides the best trade-off between stability and credit assignment for memory-management tasks."

# Part 5: DeltaMem（arXiv 2604.01560）—— RL 记忆的 critical voice

## 5.1 核心 critique

DeltaMem 是一篇 **批判性** 论文——它 explicitly 指出 RL 记忆方法（Memory-R1 / AgeMem / Mem-α）的根本问题：

> **"用 downstream QA accuracy 做 proxy reward 导致严重稀疏奖励、难收敛。"**

### 5.1.1 稀疏奖励问题的根源

```
A long dialogue:
  turn 1: model ADD memory_1
  turn 2: model ADD memory_2
  ...
  turn 100: model RETRIEVE
  turn 101: user asks question
  turn 102: model answers
  reward: 0 or 1 (basd on answer correctness)
```

问题：

- 100 个动作只有 1 个 reward 信号
- 不知道是哪个具体动作错了
- credit assignment 极难

### 5.1.2 DeltaMem 的 evidence

DeltaMem 论文里展示训练曲线：

- Memory-R1 在某些场景训练不稳定（reward 在 0.3 附近震荡）
- AgeMem 收敛但需要大量 sample
- Mem-α 在长 sequence 上 overfit short sequence

**这些 evidence 是 reviewer 会引用来质疑你方法的**。

## 5.2 DeltaMem 的解法（你应该借鉴的）

DeltaMem 提出几种 mitigation：

### 5.2.1 过程奖励（Process Reward）

不只用最终 QA，给每个 step 一个 process reward：

```python
def step_reward(action, state):
    if action.type == ADD:
        # Reward: is this content novel? relevant?
        reward = novelty(action.content) * relevance(action.content, state.context)
    elif action.type == UPDATE:
        # Reward: does this update resolve a conflict?
        reward = conflict_resolution_score(...)
    ...
```

### 5.2.2 操作级监督（Operation-level Supervision）

对每个动作有 demonstration data：

```
state: ...
expected_action: ADD("user mentioned new job", speaker_id=Alice)
```

把这种 supervision 加入 RL 训练（mix RL + behavior cloning）。

### 5.2.3 Reward shaping

给 sparse outcome reward 添加 dense intermediate reward。

## 5.3 对你的关键意义

**DeltaMem 是你 Idea 3（GroupReward）的直接 motivation**。Idea 3 的核心就是 "针对 DeltaMem 揭示的问题，构造过程奖励"。

在你的论文里：

> "Recent work [DeltaMem, 2026] explicitly criticizes outcome-only RL for memory management, showing that proxy rewards lead to severe sparsity and unstable convergence. Our work directly addresses this critique by introducing operation-level process rewards via a learned PRM..."

这种 **正面回应 critique** 是 strong 的论文写作策略。

# Part 7: 总体 Related Work 写法建议

> "The RL-based memory management literature [Memory-R1, AgeMem, Mem-α, MemSearcher] faces three known challenges: (1) sparse reward leading to unstable convergence, explicitly critiqued by DeltaMem [2026]; (2) the 'Echo Trap' phenomenon in multi-turn training [RAGEN, 2025], where policies collapse to repetitive templates; (3) credit assignment across multi-turn trajectories, addressed differently by step-wise GRPO [AgeMem], multi-context GRPO [MemSearcher], and M-GRPO [WebAgent-R1]. # Part 8: 一个 reviewer-friendly 的写作策略

AAAI 审稿人通常是 AI/ML 领域的 senior researcher，对历史经典熟悉。**引用 AAAI 自家经典**（如 SI-RNN）能：

1. **展示 historical awareness**：你不是 NLP 新手，知道领域脉络
2. **建立 credibility**：审稿人会认为你做的工作是 "AAAI 传统的延伸"
3. **减少 reviewer skepticism**：cite 经典让审稿人感到熟悉

具体的 cite 候选：

- SI-RNN (AAAI 2018) - 多方对话
- Memory Networks (NeurIPS 2014) - 记忆系统
- Knowledge Graph Embedding (AAAI 多篇) - 实体表示

**实操**：在你的 Introduction 和 Related Work 至少 cite 2-3 篇 AAAI 历史经典。这是低成本高 ROI 的 reviewer-friendly tactic.
