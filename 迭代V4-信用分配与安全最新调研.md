## Part 1：Counterfactual Credit Assignment 爆发期（2026 年 3-5 月）

### 1.1 C3：Contextual Counterfactual Credit Assignment（2603.06859，2026/3）

**全名**：Contextual Counterfactual Credit Assignment for Multi-Agent RL in LLM Collaboration  
**GitHub**：https://github.com/EIT-EAST-Lab/C3（有代码！）

**核心方法**：
- 框架：多 agent 协作中的 per-message causal credit
- 技术：**Leave-One-Out (LOO) baseline**
- 做法：冻结当前对话 transcript，替换该 agent 的特定发言（counterfactual），通过 fixed-continuation replay 评估结果变化
- 关键洞见：LLM 协作系统的 interaction history 是 observable text 的**确定性函数**（无隐藏状态），所以任何 decision point 都可以精确复原 → counterfactual 不需要额外环境交互

**LOO 公式**：
```
A_i^LOO = R(τ_original) - E[R(τ_{-i})]

其中 τ_{-i} = 用 counterfactual message 替换 agent i 发言后的轨迹
     E[...] = 在 frozen behavior policy 下 sample K 次取平均
```

**对比 V2 中 CoMAM 的 adaptive credit**：
- CoMAM：group-level ranking consistency（启发式，非因果）
- C3：真正的反事实推断（LOO，因果严格）

**对我们的价值**：
- C3 框架可以直接扩展到 **speaker-aware context**：把"message"换成"speaker 的 memory 操作"
- 具体：R(τ_{-speaker_X}) = 如果删除 speaker X 的所有 memory 操作，结果会怎样？
- 这就是 Idea 9 的精确实现方案

**SpeakerC3 扩展**：
```python
def speaker_c3_credit(trajectory, speaker_id, G=4):
    """
    计算 speaker_id 的 memory 操作对最终回答质量的因果贡献
    """
    original_reward = reward_v3(trajectory)
    
    # 反事实：删除 speaker_id 的所有 memory 操作
    counterfactual_trajectories = []
    for _ in range(G):
        # 用 frozen behavior policy 从 speaker_{-i} 起重新 rollout
        cf_traj = rerollout_without_speaker(trajectory, speaker_id)
        counterfactual_trajectories.append(cf_traj)
    
    counterfactual_baseline = mean(reward_v3(cf) for cf in counterfactual_trajectories)
    
    # LOO credit = original - counterfactual
    return original_reward - counterfactual_baseline
```

**优势 vs Idea 9 原方案**：
- 原 Idea 9 是替换 speaker attribution，C3 style 是**删除整个 speaker 的操作**
- C3 style 更干净：不需要假设"如果换成其他 speaker 会怎样"
- 计算成本：G=4 次额外 rollout per speaker，训练时间增加约 N 倍

**建议**：在 SpeakerMem-R1 中，把 C3 style 的 speaker LOO 作为 Stage 2 RL 中的 **credit validation check**（不是主要 reward，而是用于识别哪些 speaker 的操作对结果有正面贡献）。

### 1.2 CCPO：Counterfactual Credit Policy Optimization（2603.21563，2026/3）

**核心方法**：通过"移除 agent 的贡献"来构建反事实 baseline  
**GitHub**：https://github.com/bhai114/ccpo

**与 C3 的关键区别**：
- CCPO：agent removal（模拟 agent 不参与的情况）
- C3：message replacement（模拟 agent 说了不同的话的情况）
- CCPO 引入 distribution shift 问题（agent 被移除后其他 agent 的行为也会变化）
- C3 通过 frozen policy 避免了 distribution shift

**对多方记忆的应用**：
- CCPO 的"agent removal"对应我们的"speaker X 没有说话时的场景"
- 这比 C3 的 message replacement 更接近我们的实际需求（某些群聊中某个 speaker 可能缺席）

### 1.3 COSAC：Counterfactual Credit in Sequential Cooperative Teams（2604.17693，2026/4）

**核心创新**：
- **Ridge regression**：用一次批次 rollout 拟合 per-agent reward decomposition（非常轻量！）
- **Fictitious continuations**：用 policy forward pass（而非 environment replay）生成 counterfactual
- 实例化 **Sequential Aristocrat Utility（SeqAU）**

**优势**：
- 不需要额外的 environment 或 reward 调用（只需 policy forward pass）
- 计算成本：近乎零额外开销（ridge regression on rollout batch）

**对 SpeakerMem-R1 的启发**：
- COSAC 的 ridge regression decomposition 可以用于**per-speaker reward 分解**：
```
R_total ≈ w_Alice × R_alice_ops + w_Bob × R_bob_ops + ... + ε
```
- 用批次数据 fit 这个 ridge regression，得到每个 speaker 的 memory 操作的边际贡献系数

这是 MemBuilder 的贡献感知梯度加权的**更严格版本**！

### 1.4 CausalFlow：Causal Attribution + Repair（2605.25338，2026/5）

**核心创新**：
- 把 failed execution traces → 可验证的 minimal repairs
- **Causal Responsibility Scores（CRS）**：通过 step-level counterfactual intervention 识别失败诱因步骤
- 生成 contrastive pairs（wrong step, corrected step）→ 可以作为 SFT 数据！

**对 SpeakerMem-R1 的启发**：
- 当 SpeakerMem-R1 在 benchmark 上失败时，用 CausalFlow 分析**哪个 memory 操作导致了失败**
- 生成 (wrong_memory_op, correct_memory_op) pair → 加入 Stage 1 SFT 数据

这提供了一个**在线数据增强**方法：
```
Train → Evaluate → CausalFlow 分析失败 → 生成修复 pair → 加入训练集 → 继续训练
```

### 1.5 Causal Intervention-Based Memory Selection（2605.17641，2026/5）

**全名**：Causal Memory Intervention (CMI) for Long-Horizon LLM Agents

**核心创新**：
- 现有 retrieval 用 semantic similarity，但 topically similar ≠ causally relevant
- **CMI**：通过 controlled intervention 估计每条 memory 对 answer 的因果影响
  - 有这条 memory 时的答案质量 vs 没有这条 memory 时的答案质量
  - 选择 causally positive 的 memory，过滤 causally negative 的

**Causal-LoCoMo Benchmark**：
- 派生自 LoCoMo
- 每个样本有：useful memories + irrelevant distractors + synthetic harmful memories
- 标注了 causal relevance（不只是 semantic relevance）

**对 SpeakerMem-R1 的启发**：

**1. Retrieval Policy 改进**：
```python
def causal_speaker_retrieval(query, memory, speaker_context):
    """
    不用 semantic similarity 做 retrieval，
    而是用 causal intervention：
    "如果我检索 memory[s]，对最终答案的因果贡献有多大？"
    """
    scores = {}
    for speaker in memory.keys():
        # 有 speaker 记忆时的答案质量
        with_mem = answer_quality(query, memory=memory[speaker])
        # 没有 speaker 记忆时的答案质量
        without_mem = answer_quality(query, memory=empty)
        # 因果贡献 = 差值
        scores[speaker] = with_mem - without_mem
    
    # 选择因果贡献最大的 speaker 的记忆
    return memory[argmax(scores)]
```

**2. Causal-LoCoMo 可作为我们的 transfer test**：
- 我们的方法在 dyadic Causal-LoCoMo 上是否仍有竞争力？
- 多方版本：Causal-GroupMem（可以自己构建）

## Part 2：记忆安全/审计 爆发期（2026 年 5 月）

### 2.1 MemAudit（2605.23723，2026/5）

**核心创新**：
- **问题**：adversarial user 通过普通交互植入恶意 memory，后续被检索到影响 agent 行为
- **方法**：双信号融合
  1. **Counterfactual Memory Influence Score**：replay-based intervention，估计每条 memory 的因果贡献
  2. **Structural Anomaly Detector**：检测 memory graph 中语义不一致模式
- **攻击模型**：MINJA（memory injection through ordinary interaction）

**和我们的关系**：
- MemAudit 解决 post-hoc 审计（事后检测）
- 我们的 QUARANTINE 机制解决 proactive 防护（事前隔离）
- 两者互补！论文中可以说：SpeakerMem-R1 的 quarantine 提供**在线防护**，MemAudit 提供**审计机制**

### 2.2 MAGE（2605.03228，2026/5）

**全名**：Safeguarding LLM Agents against Long-Horizon Threats via Shadow Memory

**核心思路**：维护一个"shadow memory"来检测和过滤有害记忆操作。

**对我们的补充**：shadow memory = 我们的 quarantine 区域的另一种实现形式。
