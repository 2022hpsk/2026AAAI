G-Memory（arXiv 2506.07398，Guibin Zhang et al.，2025 年 6 月，NUS/同济/UCLA/A\*STAR/NTU 联合）是 **多 Agent 系统记忆** 这条线的代表作。它的核心贡献是**三层图层级记忆结构**，把多 Agent 协作中的不同抽象层级（具体协作轨迹 → 任务关联 → 跨任务可泛化洞察）显式建模。对你做"共享/私有记忆"研究，这是必须深读的工作。

## 1. "受组织记忆理论启发" —— 学科借鉴

G-Memory 的方法论根基在"组织记忆理论"（Organizational Memory Theory，Walsh & Ungson 1991 的经典）。这个理论的核心观点是：

**组织（公司、团队、部门）的记忆不是单一的，而是分层的**：
- 个体记忆（每个员工脑子里的）
- 共享文档/数据库（团队公开的）
- 制度化知识（公司层面的 SOP、文化）

不同层次的记忆服务不同的认知功能：个体记忆适合 specific task execution，共享记忆适合 cross-task coordination，制度化知识适合长期 strategic planning。

G-Memory 把这套框架搬到 LLM 多 Agent 系统：

```
Insight Graph     ↔  制度化知识     （跨任务可泛化洞察）
       ↑
Query Graph       ↔  共享文档/数据库 （task-level 关联）
       ↑
Interaction Graph ↔  个体记忆        （具体协作轨迹）
```

**对你的研究启示**：这个三层结构正好对应你多角色场景的 "个体 speaker private" / "群组 shared" / "跨群组 wisdom" 三层。组织记忆理论给你一个 well-established 的学术根基——AAAI 审稿人喜欢看到这种"有理论根基的架构设计"。

## 2. 三层图的具体定义

### 2.1 Interaction Graph（最底层）

每次多 Agent 协作完成一个 task，会产生一段交互轨迹。Interaction Graph 把这段轨迹"凝练"成一个图：

```
节点：参与协作的 Agent + 关键 sub-step
边：消息传递、依赖关系、time-ordering
```

注意是"凝练"而非"原始记录"——LLM 会从原始轨迹中提取关键交互模式，去除冗余。一个 task 完成后产生一个 interaction graph。

### 2.2 Query Graph（中间层）

把多个 interaction graph 按 task / query 类型聚类。同类 query 共享一个 query graph 节点，节点上挂着指向多个 interaction graph 的指针。

```
"如何写一个网页爬虫" Query Node
    ↓ 指向
    - Interaction Graph A（用 Selenium 完成的轨迹）
    - Interaction Graph B（用 Playwright 完成的轨迹）
    - Interaction Graph C（用 BeautifulSoup 完成的轨迹）
```

### 2.3 Insight Graph（最上层）

跨 query graph 抽象出的可泛化洞察。比如：

```
Insight 节点："对于动态加载页面，需要先 wait for JS 渲染"
    ↓ 关联到
    - Query Graph: "如何爬虫"
    - Query Graph: "如何测试 web app"
    - Query Graph: "如何 monitor 网站变化"
```

Insight 不绑定具体 task，而是 cross-domain 的"教训"。

### 2.4 三层之间的关系

```
Insight ← (抽象自) ← Query ← (聚类自) ← Interaction
```

新协作发生时：
1. 新 interaction graph 被加到底层
2. 周期性聚类，更新 query graph 的关联
3. 周期性抽象，更新 insight graph

## 3. "MAS 表为 G=(V,E)" —— 形式化

G-Memory 把多 Agent 系统形式化为：

- G = (V, E)：N 个 agent V，T 个同步通信 epoch
- 按拓扑序 π 处理 agent → 每个 epoch 内 agent 按顺序响应

这种形式化的好处是把 MAS 协作变成图问题——所有图算法（traversal, clustering, embedding）都能直接用。

**对你的多角色场景**：群聊也可以表示为图：

- 节点 = speaker
- 边 = 消息回复关系、@提及关系
- 时间维度 = epoch 序列

你可以直接借鉴 G-Memory 的形式化框架。

## 4. "Bi-directional memory traversal" —— 检索机制

G-Memory 的检索是双向的：

**Top-down traversal**：从 insight 出发
```
查询："怎么做爬虫"
1. 检索 Insight Graph 找相关洞察
2. 沿链接下到 Query Graph 找具体相关 query
3. 再下到 Interaction Graph 找具体轨迹
```

**Bottom-up traversal**：从 interaction 出发
```
查询：当前在写一个爬虫
1. 检索 Interaction Graph 找具体轨迹细节
2. 沿链接上到 Query Graph 找同类任务的其他轨迹
3. 再上到 Insight Graph 找 high-level 教训
```

**为什么双向**：单向遍历有信息丢失风险——只看 high-level 错过细节，只看 low-level 错过共性。双向同时获取多 granularity 信息。

**对你的多角色场景**：你的检索可以是 "speaker-aware bi-directional"：

```
Top-down：从群组 shared insight → speaker private episodic
Bottom-up：从某 speaker 的具体发言 → 群组 shared 共识
```

## 5. 实验结果

G-Memory 的实验非常扎实：

- **5 benchmark**：ALFWorld（embodied）、SciWorld（embodied）、PDDL（planning）、HotpotQA、FEVER（QA）
- **3 backbone**：GPT-4o-mini、Qwen2.5-7B、Qwen2.5-14B
- **3 MAS 框架**：AutoGen、DyLAN、MacNet

具体数字：

| Benchmark   | Baseline | G-Memory | 提升       |
|-------------|----------|----------|------------|
| ALFWorld    | 58.21%   | 79.10%   | +20.89%   |
| SciWorld    | -        | -        | 显著提升  |
| HotpotQA    | -        | -        | +10.12%   |
| FEVER       | -        | -        | 显著提升  |

关键消融：
- **去掉 insight graph 掉点**：5-10%
- **去掉 interaction graph 掉点**：8-15%
- **去掉 query graph 掉点**：3-5%

结论：三层都重要，interaction 层最关键（细节信息），insight 层提供跨任务转移。

**1-hop vs k-hop 扩展**：1-hop 通常最优，k∈{1,2} 之外掉点。这是因为更远的邻居引入噪声。

## 6. G-Memory 的根本 limitation —— training-free

G-Memory 是 **training-free** 的——所有的"插入新 interaction graph"、"聚类成 query graph"、"抽象成 insight graph"都用 GPT 做。这有几个明显代价：

**Limitation 1**：聚类/抽象的质量受 prompt 工程影响极大。换个 LLM 或换个 prompt 可能效果大变。

**Limitation 2**：没有 closed-loop 反馈。如果某条 insight 对下游帮助不大，G-Memory 没有机制把它"降级"或删除。

**Limitation 3**：检索策略是固定的（bi-directional traversal），不是 task-adaptive 的。

**对你的研究意义**：这些 limitation 全是你 RL 方法的卖点。具体来说：

- 你可以**用 RL 学三层之间的路由策略**——什么 interaction 该 promote 成 insight，什么 query 该被检索多少 hop。
- 你可以**用 outcome feedback 修剪低价值 insight**——长期没被检索/使用的 insight 应该降级或丢弃。

## 7. 对你做多角色研究的具体借鉴

### 7.1 对 Idea 1（SpeakerMem-R1）

借鉴：三层结构 →"speaker-grounded 三层"：

```
Speaker Insight Layer：跨 speaker 的群组共识、规范
       ↑
Speaker Query Layer：按话题聚类的 speaker 发言
       ↑
Speaker Interaction Layer：原始 speaker turn 序列
```

### 7.2 对 Idea 2（GroupPolicy）—— 这是 G-Memory 最直接的延伸

G-Memory 的三层正好映射到你的"共享 vs 私有"边界：

- Interaction Graph = **私有** 协作轨迹（细节，仅参与 agent 可见）
- Insight Graph = **共享** 提炼知识（可跨 agent 引用）
- Query Graph = 中间状态（部分共享）

**你的 delta**：把三层之间的 promote / demote 决策做成 RL 动作。具体：

- `PROMOTE(interaction_id → query_id)`：决定某条具体轨迹是否值得聚类到 query 层
- `ABSTRACT(query_ids → insight_id)`：决定从一组 query 抽象出 insight
- `DEMOTE(insight_id)`：把低价值 insight 降级
- 奖励：下游协作任务成功率 + 隐私泄露惩罚

**这就是你 Idea 2 的核心架构**——G-Memory 提供了 structure，你提供 learning。

### 7.3 对 Idea 3（GroupReward）

借鉴：三层 graph 的每一层都可以有自己的过程奖励：

- Interaction 层：交互轨迹的 fidelity（是否如实记录了协作）
- Query 层：聚类的 coherence（同 cluster 内 query 的语义一致性）
- Insight 层：abstraction quality（insight 是否真的概括了多个 query）

每层奖励作为 PRM 的子组件。

## 9. 一个关键的方法论 angle —— 把 G-Memory 当作 "structure prior"

G-Memory 提供了一个非常优秀的 **structure prior**——三层图、双向遍历、按 epoch 处理——但所有这些 structure 都是 hardcoded 的策略，不是学出来的。

你的论文的一种聪明定位是："**借用 G-Memory 的 structure prior，但把所有 promotion / retrieval / pruning 决策做成 learned policy**"。这种"hard structure + soft policy"的设计在 AAAI 是非常受欢迎的——既不是"我从零搭一个 architecture"（太 ambitious），也不是"我只 fine-tune"（太 incremental），而是"我借用 well-motivated structure，加上 learning 优化"。

具体的 Method 章节框架可以是：

1. **背景**：G-Memory 的三层图（cite + 简要描述）
2. **我的扩展**：speaker 维度的注入（每个节点带 speaker_id 标签）
3. **我的学习目标**：跨层 promote/demote 用 RL 学
4. **训练算法**：multi-context GRPO（borrowed from MemSearcher）
5. **奖励**：下游 + 过程奖励（borrowed from AgeMem）

这种结构清晰、归属诚实、贡献明确，是 AAAI 论文的标准范式。

## 10. 一个 caveat —— G-Memory 是 multi-agent 的，你的场景是 multi-party dialogue

要明确两者的区别：

- **G-Memory 的 multi-agent**：多个 LLM agent 协作完成一个 task。每个 agent 有自己的 role（manager / executor / critic）。
- **你的 multi-party dialogue**：多个 human user 在群聊里对话，一个 LLM 在中间做记忆和回复。

二者的不同之处：

| 维度       | G-Memory MAS         | 你的多角色场景        |
|------------|----------------------|----------------------|
| 参与者     | 多个 LLM agent       | 多个 human user      |
| 协作目标   | 完成 task            | 自然对话+ad-hoc 提问 |
| 沟通模式   | 结构化（按 topo 序）  | 自由（多线程）       |
| Speaker 数 | 通常 3-10            | 通常 5-30+           |

这意味着你不能直接套 G-Memory 的代码或评测，需要做适配。但**三层结构和双向遍历的思想可以直接迁移**。你应该在论文里明确说"我们借鉴 G-Memory 的结构启发，但将其适配到 human multi-party 设置"——这种诚实归属审稿人会喜欢。
