Collaborative Memory（arXiv 2505.18279，Rezazadeh et al.，Accenture，2025 年 5 月）是 **共享/私有记忆**这条线的旗舰工作。它的核心创新是把"多用户多 Agent 环境下的记忆访问控制"做成一阶问题，并提出 bipartite access control 图作为形式化工具。**对你的 Idea 2（GroupPolicy）来说，这是直接的起点工作**，你必须细读、必须比较、必须做 delta。

## 1. "多用户多 Agent 环境" —— 问题设定

先把 Collaborative Memory 的应用场景讲清楚——它不是单用户和单 AI 助手，而是：

- 多个 user U = {u_1, u_2, ..., u_M}
- 多个 agent A = {a_1, a_2, ..., a_N}  （注：这里 agent 是 LLM agent，不是 user）
- 多个 resource R = {r_1, r_2, ..., r_K}（如文档、API、tool）

**典型场景**：

- 公司里的协作 AI 助手：法务 user 用法务 agent，财务 user 用财务 agent，CTO 用 universal agent；不同 user 看不同的文档（resource）。
- 多用户共享 AI 知识库：team 内部共享某些笔记，个人项目笔记私有，跨 team 项目笔记 selectively 共享。

**核心张力**：信息怎么在不同 user/agent 之间安全地流动？

- 张三在和法务 agent 聊新合同 → 学到的知识能用在李四的合同 review 里吗？
- 张三和销售 agent 聊客户 X 的偏好 → 这个偏好能被市场 agent 用来设计 campaign 吗？

这种**跨 user 跨 agent 的记忆共享**就是 Collaborative Memory 要解决的问题。

## 2. "非对称、时变的 access control" —— 关键性质

权限控制有几个关键性质，每个都影响记忆系统设计：

### 2.1 非对称（Asymmetric）

A 能看 B 的某些信息，不意味着 B 能看 A 的对应信息。比如：

- Manager 能看下属的工时，下属看不到 manager 的薪资。
- 客户能看 AI 助手为他生成的报告，但看不到 AI 助手生成报告时调用了哪些内部工具。

### 2.2 时变（Time-varying）

权限会随时间变化：

- 新员工入职 → 获得某些权限
- 离职 → 撤销所有权限
- 临时项目 → 短期获得跨部门权限

时变意味着 "访问决策必须在每次访问时实时检查"，不能 cache。

### 2.3 多对多

一个 user 可能用多个 agent，一个 agent 可能服务多个 user，一个 resource 可能被多个 (user, agent) 对组合访问。这是 bipartite/tripartite 关系。

## 3. "Bipartite 图建模" —— 形式化

Collaborative Memory 用一个 bipartite 图（或 tripartite，看怎么定义）来表示权限：

```
节点：U（users）+ A（agents）+ R（resources）
边：
  - U-A 边：user u 可以使用 agent a
  - A-R 边：agent a 可以访问 resource r
  - U-R 边：user u 可以访问 resource r（独立于 agent）
```

权限的计算：

```python
def can_access(user, agent, resource, timestamp):
    has_U_A_edge = check(user, agent, timestamp)
    has_A_R_edge = check(agent, resource, timestamp)
    has_U_R_edge = check(user, resource, timestamp)
    
    # 必须 user 能用 agent，且 agent 能访问 resource，且 user 自己也有该 resource 权限
    return has_U_A_edge and has_A_R_edge and has_U_R_edge
```

**对你做多角色场景的启示**：你的群聊场景有类似但更简单的访问图：

```
节点：speakers + group_channels + AI_assistant
边：
  - speaker-channel 边：speaker 是 channel 的成员
  - speaker-AI 边：speaker 可以和 AI 对话
  - channel-AI 边：AI 在某 channel 中活跃
```

每个 message 都有 (speaker, channel, timestamp) 元数据，定义了它的访问范围。

## 4. "Private vs Shared 双层记忆" —— 核心架构

Collaborative Memory 的核心架构是**双层记忆**：

### 4.1 Private Memory

每个 user 一份独立的 private memory：

```python
private_memory[user_id] = {
    fragments: [
        {content, provenance: (agent_id, resource_ids, timestamp), ...},
        ...
    ]
}
```

**只有创建该 fragment 的 user 可见**——即使 fragment 内容对其他 user 也有用，也不会被其他 user 看到。

### 4.2 Shared Memory

被显式标记为"可共享"的 fragment 进入 shared memory。每个 fragment 仍带 provenance，但可被多个 user 访问（受 access control 约束）。

### 4.3 双层之间的关系

新对话 turn 产生 fragment 时，**write policy 决定它进 private 还是 shared**：

```python
def write_policy(fragment, user, agent, context):
    # Collaborative Memory 的策略是基于规则
    if fragment.is_sensitive(user):
        return PRIVATE
    if fragment.has_general_value() and agent.is_sharing_enabled():
        return SHARED
    return PRIVATE  # default
```

注意 **这个 write policy 是 hardcoded 的规则**——这是 Collaborative Memory 的 limitation，也是你的 delta 入口。

## 5. "Provenance 不可变属性" —— 关键 metadata

每个 fragment 带不可变（immutable）的 provenance：

- **contributing_agent**：哪个 agent 创建了它
- **accessed_resources**：创建时用到的 resources
- **timestamp**：创建时间戳

为什么不可变：

1. **审计需求**：能追溯每条记忆的来源
2. **隐私需求**：用户能问"这个信息是从哪来的"
3. **去重**：相同内容但不同来源的 fragment 不应该合并

**对你做多角色场景的直接启示**：你的 fragment 必须带 `speaker_id` provenance——"这条信息是谁说的"。这正好是 GroupMemBench 强调的 speaker-grounded 维度。

**扩展的 provenance schema**：

```python
fragment.provenance = {
    speaker_id: str,         # 谁说的（必需）
    addressee_ids: List[str], # 对谁说的（多方场景必需）
    channel_id: str,          # 在哪说的
    timestamp: int,
    contributing_agent: str,  # 如果是 AI agent 处理过，是谁
    derived_from: List[str]   # 如果是从其他 fragments 推断的，源头是什么
}
```

## 6. "Read Policy 动态构造记忆 view" —— 读取机制

当用户 query 时，read policy 根据当前权限动态构造可见的 memory view：

```python
def construct_memory_view(user, agent, query, timestamp):
    # 1. 从 user 的 private memory 取相关 fragments
    private_relevant = retrieve(private_memory[user], query)
    
    # 2. 从 shared memory 取相关 fragments，并 filter 掉无权限访问的
    shared_relevant = retrieve(shared_memory, query)
    shared_filtered = [
        f for f in shared_relevant
        if can_access(user, agent, f.provenance.accessed_resources, timestamp)
    ]
    
    # 3. 合并、排序、限制 budget
    view = (private_relevant + shared_filtered)[:budget]
    return view
```

**关键**：每次 query 都重新构造 view，不 cache。这是因为权限时变。

**对多角色场景的应用**：群聊里，AI 助手对某 speaker 提问的回答，应该只 retrieve 该 speaker **被允许看到的** memory：

```python
def gather_memory_for_asker(asker, query):
    # 1. asker 的私有记忆
    asker_private = retrieve(private_memory[asker], query)
    
    # 2. 群组共享记忆中 asker 有权限的部分
    group_shared = retrieve(group_memory, query)
    
    # 3. 其他 speakers 的私有记忆里被 asker 提及/相关的部分（特殊权限）
    cross_speaker = retrieve_cross_speaker(asker, query, permissions)
    
    return merge(asker_private, group_shared, cross_speaker)
```

## 7. "全部组件用 GPT-4o" —— 注意 training-free 性质

Collaborative Memory 全部用 GPT-4o 做 write/read policy 决策——**完全 training-free**。这是你的 Idea 2 最直接的 delta：

- Collaborative Memory：write policy 是 GPT-4o + 规则（fixed）
- 你的 Idea 2：write policy 是 RL 学的 π_θ

具体 RL formulation：

```python
# Collaborative Memory: 固定规则
action = if rule_sensitive(fragment): PRIVATE else SHARED

# 你的 RL 方法
state = (fragment_content, current_private_memory, current_shared_memory, user_context)
action ~ π_θ(action | state)  # 学到的策略
reward = R_collab(downstream_task_success) - η * R_leak(privacy_breach) + μ * R_efficiency
```

奖励里的 **R_leak（隐私泄露惩罚）** 是关键——它需要一个 oracle 判断"这条信息是否应该 private 但被 promoted 到 shared"。Collaborative Memory 因为是 training-free 没有这个问题，你做 RL 必须设计 R_leak 的度量。

## 8. R_leak 怎么定义？

这是你 Idea 2 的核心实验设计难题。几个可能的方案：

### 方案 A：合成数据 + 显式 sensitive label

```python
# 在合成数据时，每个 fragment 标 sensitivity_level
fragments = [
    {"content": "我的 SSN 是 ...", "sensitivity": HIGH},
    {"content": "我喜欢拿铁", "sensitivity": LOW},
    ...
]

# R_leak = 模型把 HIGH sensitive 的 fragment 写入 shared 的频率
R_leak = sum(1 for f in shared_memory if f.sensitivity == HIGH)
```

### 方案 B：cross-user query 测试

```python
# 训练后，让 user B 询问关于 user A 的信息
# 如果 B 能查到 A 私密信息 → R_leak += 1
```

### 方案 C：LLM judge

```python
# 用 GPT-4 判断每个 shared fragment 是否"应该 private"
R_leak = avg(judge.is_should_be_private(f) for f in shared_memory)
```

每种方案有 trade-off。AAAI 论文里最稳妥是组合两种：合成数据（A）做主指标 + LLM judge（C）做辅助验证。

## 10. Limitation 与反驳

**Limitation 1：双层是 hardcoded 的**。即使 write policy 学了，"只有 private/shared 两层"还是固定的。你可以考虑多层（如 G-Memory 三层）。

**Limitation 2：bipartite 图本身是 input**——Collaborative Memory 不学权限图，只在给定权限下做记忆操作。**对你来说**：权限是 input 还是 learn 的边界？AAAI 论文里建议保守——权限是 input，记忆操作是 learn。

**Limitation 3：评测主要展示功能性，不强 quantitative**。**对你来说**：补 quantitative benchmark 是论文质量的关键。

## 11. 把它们串起来看：Collaborative Memory 的整体设计哲学

Collaborative Memory 的设计哲学是：**多 user/agent 协作天然需要 access control → 用 bipartite 图形式化 → private/shared 双层记忆承载边界 → provenance 保留可追溯性 → write/read policy 解耦**。

这是一个非常 production-aware 的设计——它考虑了真实企业场景的合规、审计、隐私需求，比纯学术 benchmark 工作更接地气。

**对你的研究**：你应该 **保留它的所有 architectural decision，只把 write policy 从固定规则替换为 learned policy**。这种"窄而深"的 delta 比"宽而浅"更容易在 AAAI 通过——审稿人能清楚看到"你改了什么，没改什么"。
