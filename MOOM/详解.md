MOOM（Maintenance, Organization and Optimization of Memory in Ultra-Long Role-Playing Dialogues，arXiv 2509.11860，Weishu Chen et al.，2025 年 9 月）是 **角色扮演长对话记忆**这条线的代表作。它的两个核心贡献——**双分支记忆架构（情节/人物）和竞争-抑制遗忘机制**——对你做多角色记忆有非常具体的借鉴价值。特别是遗忘机制部分，是当前所有 RL 记忆论文里最有理论根基的设计。

## 1. "首个双分支记忆插件" —— 文学理论根基

MOOM 的方法论根基不是认知科学（像 Mem-α 那样借用 Tulving），而是**文学理论**——具体说是叙事学的"情节"和"人物"二元结构。

### 1.1 文学理论的二元结构

经典叙事学（E.M. Forster, Aspects of the Novel）把小说分为：

- **情节（Plot）**：事件、冲突、转折、解决
- **人物（Character）**：性格、动机、关系、成长

一部好的小说既需要 compelling plot（让读者想知道接下来发生什么），也需要 well-developed characters（让读者关心这些人）。

### 1.2 MOOM 的映射

```
Branch 1: Plot Memory（情节记忆）
  - 记录冲突、事件、转折
  - 按多时间尺度组织（最近一场冲突 / 本卷主线 / 整部剧情）

Branch 2: Character Memory（人物记忆）
  - 记录用户/角色 profile
  - 按预定义关键词抽取（性格、偏好、目标、关系）
```

这种二元结构和 Mem-α 的 core/episodic/semantic 三元结构不冲突——可以看作不同 axis 的切分：

- Mem-α 的轴：稳定度（core 最稳定）
- MOOM 的轴：信息类型（plot vs character）

**对你的应用启示**：你的多角色场景可以同时用两个轴：

```
            Plot (event)            Character (profile)
Core        群组关键事件、决议        每个 speaker 的核心 identity
Episodic    具体 turn 序列            speaker 在某 session 的态度
Semantic    抽象的群组动态            speaker 的长期变化
```

这种 2D 切分让记忆架构更细致、更好检索。

## 2. "多时间尺度的情节总结" —— Plot Branch 细节

Plot Branch 不是 single-level，而是多层级：

```python
class PlotMemory:
    immediate: List[Event]    # 最近 5-10 turn 的具体事件
    short_term: List[Summary] # 最近 50 turn 的事件总结
    medium_term: List[Arc]    # 跨多 session 的情节弧
    long_term: List[Theme]    # 整体主题/中心冲突
```

每个层级有不同的写入条件：

- `immediate`：每 turn 直接写
- `short_term`：每 10 turn 由 LLM 总结
- `medium_term`：每 session 结束由 LLM 提取 arc
- `long_term`：手动 / 周期性 review

### 2.1 为什么多时间尺度

如果只有 single-level plot memory，存在两个问题：

1. **粒度不匹配**：用户问"这场对话谁先发起的冲突"需要 immediate 级；问"这个角色的整体动机"需要 long-term 级。single-level 无法兼顾。
2. **检索效率**：所有 plot 都在一个池子里，长对话后检索 noise 极大。

多时间尺度让检索可以按 query 粒度选层级——细节问题查 immediate，主题问题查 long-term。

### 2.2 对你做多角色场景的应用

多角色场景天然有多时间尺度：

- **immediate**：最近几个 turn 的 raw 对话
- **session**：一次群聊会话的总结
- **thread**：跨多 session 但围绕一个话题的内容
- **long-term**：整个群组的历史、关系演化

你的 RL 方法应该学**何时把 immediate 推送到 session 层**、**何时把 session 推送到 thread 层**——这些是 promotion decisions，是 RL 动作空间的一部分。

## 3. "Character Branch + 预定义关键词" —— 用户 profile 构建

Character Branch 用**预定义关键词**抽取 user/角色 profile：

```python
predefined_keywords = [
    "personality",       # 性格
    "preferences",       # 偏好
    "goals",            # 目标
    "background",       # 背景
    "relationships",    # 关系
    "current_mood",     # 当前情绪
    "writing_style",    # 表达风格
]

def update_character_memory(new_turn, character_memory):
    for keyword in predefined_keywords:
        new_info = LLM.extract(new_turn, regarding=keyword)
        if new_info:
            character_memory[keyword] = LLM.merge(
                character_memory[keyword],
                new_info
            )
```

这种 keyword-based 抽取的好处：

1. **结构化** —— 不是 free text，可以直接 query 特定字段
2. **可控** —— 通过调整 keyword list 控制抽取深度
3. **可追溯** —— 每个字段的更新可以记录 history

### 3.1 对你做多角色场景的应用

每个 speaker 维护一份 keyword-based profile：

```python
profile[speaker_id] = {
    "role": ...,             # 在群组中的角色（如 leader、observer）
    "expertise": ...,        # 专业领域
    "communication_style": ...,
    "current_topic_interest": ...,
    "relationships": {       # 与其他 speaker 的关系
        other_speaker_1: "close_friend",
        other_speaker_2: "rival",
    }
}
```

这种 per-speaker profile 比扁平 memory 更适合 speaker-specific queries。

## 4. "竞争-抑制遗忘机制" —— 核心理论贡献

这是 MOOM 最有意思的部分，**也是对你 RL 方法最有价值的借鉴**。

### 4.1 心理学根基 —— Retrieval-Induced Forgetting

竞争-抑制源自心理学的 **Retrieval-Induced Forgetting (RIF)** 理论（Anderson et al., 1994）：

**当我们 retrieve 某条记忆时，与之竞争的相关记忆会被抑制（活性降低）。**

具体例子：你回忆"狗的常见品种"，想到"金毛"、"哈士奇"、"贵宾犬"。每次 retrieve 这些，**其他没被 retrieve 的品种（如"柴犬"）的活性会下降**——这就是 RIF。

### 4.2 MOOM 的实现

```python
class MemoryWithCompetition:
    fragments: List[Fragment]
    
    def retrieve(self, query, top_k=5):
        # 计算相关性
        scores = [similarity(query, f) for f in self.fragments]
        retrieved = top_k_by(self.fragments, scores)
        
        # 关键：抑制相关但未被 retrieve 的 fragment
        for f in self.fragments:
            if f in retrieved:
                f.activation = min(1.0, f.activation + 0.1)  # 强化
            elif similarity(query, f) > THRESHOLD and f not in retrieved:
                f.activation = max(0, f.activation - 0.05)   # 抑制
        
        # 当 activation 低于阈值，标记 forget
        self.fragments = [f for f in self.fragments if f.activation > FORGET_THRESHOLD]
        
        return retrieved
```

### 4.3 为什么这种 forgetting 比艾宾浩斯曲线更好

MemoryBank 用的艾宾浩斯曲线是 **time-based decay**——记忆随时间衰减，被访问就重置。

MOOM 的 RIF 是 **competition-based**——记忆因为有"更强的竞争对手"被抑制，不只是时间过去。

具体场景对比：

- 场景：用户每天都被问关于"工作"的事，从来不被问"爱好"
- **艾宾浩斯下**：工作记忆经常被强化（重置 timer），爱好记忆慢慢衰减
- **MOOM 下**：工作记忆每次被 retrieve 都抑制相关的爱好记忆，**爱好记忆被更激进地丢弃**

MOOM 的好处是**自然形成 "topical focusing"**——记忆库会自动聚焦到 user 最常 query 的话题。

### 4.4 对你做多角色场景的关键应用

这是你 RL 方法的 **理论 prior**——RL 应该学到与 MOOM RIF 相似的策略：

```python
# 你的 RL 动作里应该有
ACTION_TYPES = [
    "ADD",
    "UPDATE",
    "DELETE",
    "SUPPRESS",  # 不删除，但降低活性（MOOM 启发）
    "BOOST",     # 强化某条记忆（被 retrieve 时自动触发）
]
```

更进一步，你可以把 **RIF 作为 RL 的过程奖励 prior**：

```python
# 当模型选择 DELETE 一个长期未被 boost 的低活性 fragment
# 给一个小正奖励（符合 RIF 直觉）

R_forgetting_prior = 1 if (deleted.activation < threshold and 
                            deleted.last_boost > T_ago) else 0
```

这种 **theory-informed reward shaping** 在 AAAI 是非常有说服力的——它给 RL 一个 well-grounded prior，避免完全黑盒。

## 5. "ZH-4O 数据集" —— 中文超长角色扮演

MOOM 配套发布了 ZH-4O 数据集：

- **平均 600 turn / 对话**
- **中文** —— 填补中文长对话数据空白
- **人工标注记忆** —— gold memory 可作 supervision
- **角色扮演场景** —— 用户和 AI 扮演不同角色，长期互动

### 5.1 这个数据集对你的价值

1. **OOD 测试集** —— 在中文 / 角色扮演场景下验证你的方法
2. **gold memory supervision** —— 训练 PRM 时有现成 ground-truth
3. **600 turn 规模** —— 验证你方法的 long-horizon 能力

但注意：ZH-4O 是 **dyadic（用户 vs AI）** 的，不是多 speaker 的。所以它适合作为 dyadic baseline 对照，不是主 benchmark。

## 6. 实验结果

MOOM 主要 metric 和性能：

| Method            | Memory Quality | Recall | LLM Calls/turn |
|-------------------|----------------|--------|----------------|
| Mem0              | 0.72           | 0.65   | 3.2            |
| MemoryBank        | 0.68           | 0.61   | 2.8            |
| A-MEM             | 0.74           | 0.69   | 3.5            |
| MOOM              | 0.83           | 0.77   | 2.1            |

关键：

- **MOOM 性能最高**（+9% vs Mem0）
- **LLM calls 最少**（-34% vs A-MEM）—— 因为竞争-抑制减少了无效记忆维护

这个 "性能更高 + 调用更少" 的双重叙事是 production 友好的。

## 7. Limitation 与反驳

**Limitation 1：双分支是 hardcoded 的**。plot/character 二分法可能不适用于所有场景（比如纯任务对话）。**对你的意义**：你不需要照搬这个二分，可以用更适合多角色场景的切分。

**Limitation 2：keyword 预定义**。Character Branch 用固定 keyword 抽取，新出现的 character traits 可能漏掉。**对你的意义**：你的方法应该用 learned attention 而不是 fixed keyword。

**Limitation 3：竞争-抑制是 inference-time 的，没有 training**。MOOM 的 forgetting 是 inference 时计算，不是 learned。**对你的意义**：你的 RL 可以 learn 何时该 SUPPRESS——把 MOOM 的固定阈值变成 learned threshold。

## 8. 把它们串起来看：MOOM 的整体设计哲学

MOOM 的设计逻辑是：**借用文学理论的 plot/character 二分 → 各分支独立组织 → 用 RIF 启发的竞争-抑制控制无界增长 → 在中文超长角色扮演场景验证**。

这个哲学的核心是 **"借用 well-established theory + 工程实现"**——它不发明新理论，而是把 RIF 这种心理学成熟理论引入 LLM 记忆。这种思路 AAAI 审稿人非常欣赏，因为它有 theoretical grounding。

## 10. Related Work 写法建议

## 11. 一个 critical 的方法论 angle

MOOM 提出的"竞争-抑制"机制虽然在它的 dyadic 角色扮演场景下 work，但在 **多角色场景** 下需要做扩展：

- Dyadic：用户和 AI 之间的 query 序列。竞争发生在所有记忆 fragment 之间。
- Multi-party：多个 speaker，每个 speaker 都可能 query。**竞争应该是 per-speaker 的**——speaker A 频繁查询某话题，应该抑制 speaker A 的其他话题记忆，但**不应该抑制 speaker B 的对应记忆**。

这种 **per-speaker competition** 是你做多角色 forgetting 的关键 delta。在论文里你可以这样定位：

> "We extend MOOM's competition-inhibition forgetting to multi-party settings by introducing per-speaker competition pools, where retrievals by speaker A only suppress memories within A's competition pool, preserving cross-speaker memory diversity..."

这是一个有清晰技术 delta 的扩展点，AAAI 审稿人会喜欢。
