## 1. 一句话总结

HiPER 把 LLM agent 策略分为"高层规划者（出 subgoal）"和"低层执行者（完成 subgoal）"，用**分层优势估计（HAE）**在两个层次上分别赋予信用，解决稀疏奖励下长程任务的训练问题，在 ALFWorld 达到 97.4%、WebShop 达到 83.3%（Qwen2.5-7B）。

---

## 2. 背景与动机

### 2.1 问题：平面 RL 策略的局限

传统 LLM agent 用"平面（flat）"策略：在每步直接输出 action，policy 必须跨越整条轨迹传播信用：

```
问题 → step 1 action → step 2 action → ... → step 30 action → reward
         ↑                                                        │
         └─────────────────── credit 很难传回 ────────────────────┘
```

在 30 步的长程任务中，step 1 的 action 对最终 reward 有多大贡献？传统 GRPO 无法精确回答。

### 2.2 HiPER 的分层思想

把任务分解为两级：

```
高层：目标分解（long-horizon planning）
  step 1: 把"完成复杂任务"拆成 subgoal G1, G2, G3
  step 2: 输出 subgoal G1 = "找到红色杯子"
  
低层：执行（short-horizon execution）  
  step 1 of G1: "向左移动"
  step 2 of G1: "拾起红色杯子"
  ↓ subgoal G1 完成（获得局部 reward）
  
  step 1 of G2: ...
```

**关键**：每个 subgoal 完成时，可以获得**局部奖励**——即使最终任务未完成，完成某个 subgoal 也有价值。这大幅稠密化了奖励信号。

## 3. 技术方法：HAE（分层优势估计）

### 3.1 两级策略

```python
class HiPERPolicy:
    def __init__(self, llm):
        # 同一个 LLM，但用不同 prompt 扮演两个角色
        self.high_level_planner = HighLevelPlanner(llm)  # 产生 subgoal
        self.low_level_executor = LowLevelExecutor(llm)  # 执行 subgoal
    
    def forward(self, state):
        # 高层：决定当前 subgoal
        subgoal = self.high_level_planner(state)
        
        # 低层：执行 subgoal（多步 micro-actions）
        micro_actions = []
        while not subgoal_complete(state, subgoal):
            action = self.low_level_executor(state, subgoal)
            state = env.step(action)
            micro_actions.append(action)
        
        return subgoal, micro_actions
```

### 3.2 分层优势估计（HAE）

**高层优势**（评估 subgoal 选择是否好）：
```
Â_high(G_t) = R_task(final) + sum(R_subgoal(G_k) for k > t) - V_high(state_t)
```

**低层优势**（评估 micro-action 是否好）：
```
Â_low(a_t | G_t) = R_subgoal(G_t) - V_low(state_t, G_t)
```

**直觉**：
- 高层 policy：如果选了好的 subgoal，高层优势高
- 低层 policy：如果在给定 subgoal 下执行得好，低层优势高
- 两者独立更新，互不干扰

## 4. 与我们 SpeakerMem-R1 的关系

### 4.1 分层 memory 策略的类比

HiPER 的两层结构天然对应我们的 memory 管理策略：

| HiPER 层级 | 功能 | 我们的对应 |
|-----------|------|-----------|
| High-level planner | 决定当前 subgoal（写哪层？写哪个 speaker？） | Memory routing policy |
| Low-level executor | 具体操作（WRITE/UPDATE/DELETE 细节） | Memory operation policy |

**具体对应**：

```
高层决策（high-level planner）：
  "当前 turn 来自 Alice，内容是个人经历 → 应该写入 per_speaker_episodic[Alice]"
  vs "内容是群组讨论的决策 → 应该写入 group_insight"
  
低层执行（low-level executor）：
  "写入 per_speaker_episodic[Alice] → 具体内容如何表述？有无需要 UPDATE 已有记录？"
```

### 4.2 对 memory routing 的启发

GroupMemBench 的一个核心挑战是：AI 助手需要知道把信息存到哪里（per-speaker vs group）。这正是高层规划问题！

```python
class SpeakerMemHiPER:
    """
    把 HiPER 的分层思想应用到 speaker-aware memory 管理
    """
    
    def high_level_route(self, turn, current_memory):
        """
        高层决策：这条 turn 应该如何处理？
        输出 (target_layer, target_speaker, operation_type)
        """
        prompt = f"""
对话 turn：{turn.text}
说话人：{turn.speaker}
当前群聊成员：{turn.participants}

请决定：这条信息最适合存储在哪一层记忆中？
选项：
1. per_speaker_core[{turn.speaker}]  - 核心个人事实
2. per_speaker_episodic[{turn.speaker}] - 情节记忆
3. group_interaction  - 群组互动
4. group_insight  - 群组洞察
5. NOOP  - 不值得存储

输出格式：<layer>|<reason>
        """
        return self.high_level_planner(prompt)
    
    def low_level_operate(self, turn, target_layer, current_memory):
        """
        低层执行：具体如何写/更新 target_layer 中的内容？
        """
        existing_fragments = current_memory.get_layer(target_layer)
        
        prompt = f"""
需要向 {target_layer} 中添加/更新信息。
已有记忆：{existing_fragments}
新的 turn：{turn.text}

请执行最合适的操作：
- WRITE(新内容)：如果是全新信息
- UPDATE(fragment_id, 新内容)：如果有已有记录需要更新
- NOOP：如果已有记录已经足够
        """
        return self.low_level_executor(prompt)
```

### 4.3 HAE 在多方场景的扩展

**传统 HAE 的 subgoal = task 分解**（把"完成购物任务"分成"找商品"、"加入购物车"等）

**我们的 HAE 扩展 = memory 分层决策**：
- 高层 advantage：routing 决策是否正确（是否把 Alice 的信息存到了 Alice 的层？）
- 低层 advantage：具体操作是否正确（写入的内容是否准确？是否需要 UPDATE？）


## 5. 技术定位

### 5.1 HiPER 与其他 credit assignment 方案的比较

| 方案 | 解决的问题 | 在多方 memory 的适用性 |
|------|-----------|----------------------|
| GRPO | 无（flat） | 差（credit 无法传播到 routing 层） |
| AgeMem step-wise | 时间切片 | 可以，但不区分 routing/operation |
| Memory-R2 LoGo | 跨轨迹公平比较 | 可以，但不分层 |
| Mem-T MoT | 操作级树搜索 | 可以，但不分层 |
| HCAPO | hindsight 单操作 | 可以，但不分层 |
| HiPER HAE | **分层** routing + operation | **最适合 memory routing** |
| **我们** | 分层 × speaker-aware | 多方场景下最优 |

### 5.2 计算成本

HiPER 相比 flat GRPO 的额外开销：
- 需要 reward function 能区分 subgoal completion（需要设计局部 reward）
- 两个 policy update pass（高层 + 低层），约 1.5× 标准 GRPO

我们的 memory routing + operation 分层：
- 局部 reward = routing 正确性（speaker attribution F1）
- 可以在 Stage 3 引入，Stage 2 用 flat warm start

## 6. 如何在 SpeakerMem-R1 中使用 HiPER 思想

### 6.1 作为架构选择

我们可以在以下两个架构中二选一：
1. **Flat**（当前 v3 设计）：一个统一 policy，直接生成 8 类 action（含 speaker/audience 参数）
2. **Hierarchical（HiPER 风格）**：
   - High-level：routing policy（决定写哪层/写哪个 speaker）
   - Low-level：operation policy（决定 WRITE/UPDATE/DELETE 细节）

**推荐**：V3 暂用 Flat（更简单），作为 ablation 选项提供 HiPER 风格变体。

### 6.2 作为 ablation 维度

> Ablation: `flat policy` vs `hierarchical policy (HiPER-style)`  
> 研究问题：两层决策分离是否有助于学习 speaker-aware routing？

## 7. Related Work 写法

> 分层强化学习为长程稀疏奖励问题提供了一个自然的解决框架 [cite Sutton options]。HiPER [cite] 将 LLM agent 策略分为高层规划（subgoal 生成）和低层执行（micro-action），通过分层优势估计（HAE）在两个层次分别分配信用，在 ALFWorld (97.4%) 和 WebShop (83.3%) 上取得了 SOTA。在多方对话记忆管理中，存在天然的两层决策：决定把信息路由到哪个 speaker/层的"routing 决策"，以及如何具体写入/更新的"operation 决策"。本文将 HAE 扩展到多方 speaker-aware 记忆管理，用 speaker attribution reward 作为高层 routing 决策的局部奖励，实现两层 credit 的分别优化。

## 8. 总结

HiPER 用分层策略 + 分层优势估计解决了 flat RL 的 credit propagation 难题，在标准 benchmark 上取得了 SOTA 性能。

对我们的价值：
1. **架构思路**：memory routing（高层）+ memory operation（低层）的自然两层分解
2. **局部 reward 设计**：speaker attribution F1 可作为 routing 层的局部 reward
3. **ablation 选项**：flat vs hierarchical policy 是个有趣的消融维度
4. **叙事**：我们的 5 层 memory 结构 + speaker-conditioned action 也体现了分层设计哲学
