EverMemBench（arXiv 2602.01313，2026 年 2 月）是**多方协作长程记忆**评测的另一座大山。和 GroupMemBench 互补——GroupMemBench 重在 query 多样性 + speaker grounding，EverMemBench 重在 **超长程（百万 token）+ 多群组并行**。两者都用对你做多角色研究是必报的 benchmark。我把它逐层拆开。

## 1. "1M+ token、多方多群组" —— 规模震撼

EverMemBench 的最显著特征是 **scale**：

- **每个评测实例 1M+ token**：远超 LOCOMO 的 26k token、LongMemEval 的 100k token
- **多群组**：一个 user 同时参与多个群（工作群、家庭群、兴趣群）
- **cross-topic 交织**：同一时间段不同群里讨论不同话题
- **时序演化**：跨多周/多月的时间线

### 1.1 为什么这种 scale 重要

真实场景里，user 的群聊不是孤立的——他可能：

- 工作群里讨论项目 A
- 同时家庭群里讨论周末计划
- 同时兴趣群里讨论一个游戏
- 一周后他在工作群提到 "周末我们家去了 X"，引用了家庭群的内容

记忆系统要能 **跨群组关联** 这些信息，并支持 **超长时间跨度** 的回忆。

### 1.2 对你做记忆研究的难度

1M+ token 直接挑战记忆系统的 scalability：

- 检索效率：1M token 的 embedding 索引非常大
- 记忆管理：fragments 数量轻松达到几万条
- 评测时间：每个 query 的延迟可能几秒

**你的 RL 方法在 EverMemBench 上 work 的关键是 token-efficient design**——MemAgent / MemSearcher 的 near-constant memory 思路在这里特别重要。

## 2. "三阶段合成 pipeline" —— 数据生成

EverMemBench 的数据合成非常精细，分三阶段：

### Stage 1: Pipeline 设计

LLM 生成一个 "evergreen project" 的 high-level 计划：

```
Project: 公司全年 OKR
  Q1: ...
  Q2: ...
  Q3: ...
  Q4: ...
  
Stakeholders:
  - Alice (CEO)
  - Bob (CTO)
  - Charlie (PM)
  - ...

Group Communications:
  - Leadership group (Alice, Bob, Charlie)
  - Engineering group (Bob, David, Eve)
  - Marketing group (Alice, Frank, Grace)
  ...
```

### Stage 2: 时间线增量合成

按项目计划 + 公开新闻（带日期）增量合成 conversations：

```python
for date in project_timeline:
    # 检查日期是否有 newsworthy events（用于真实感）
    news = lookup_real_news(date)
    
    # 检查日期是否有项目 milestone
    milestone = check_project_milestone(date)
    
    # 对每个 group 生成当天对话
    for group in groups:
        if group.is_active_on(date):
            conversation = LLM.generate(
                participants=group.members,
                context=news + milestone,
                date=date,
                previous_history=group.history
            )
            group.append(conversation)
```

**关键设计**：用"真实日历对齐的公开新闻"作为外部锚点，让对话有真实感。"上周 OpenAI 发布了 GPT-X，所以我们要…"这种对话有 grounded reality。

### Stage 3: QA 生成 + 三段质控

每个对话片段配 evidence-grounded QA：

```python
def generate_qa(conversation, evidence_turns):
    qa = LLM.generate_qa(
        evidence=evidence_turns,
        question_type=sample_type()  # 见下文三维度
    )
    
    # 三段质控
    # 控制 1：LLM 盲测（无 evidence 答不出来才算合理）
    if LLM_blind_test(qa).is_answerable:
        return None  # 太简单，丢弃
    
    # 控制 2：可答性 + 唯一性
    if not check_answerability_uniqueness(qa, conversation):
        return None
    
    # 控制 3：人工审计 sample
    if random() < 0.05:  # 5% sample for human audit
        human_quality = human_audit(qa)
        if human_quality < threshold:
            return None
    
    return qa
```

这种三段质控是 **state-of-the-art 的数据质量保证**——既有 LLM 自动 filter，又有人工 spot check。

## 3. 三维度评测

EverMemBench 把 query 分为三个维度（注意和 GroupMemBench 的六类不同）：

### 3.1 Fine-grained recall

具体细节的回忆：

- "Bob 在 3 月 5 日的会议上提到的客户名是什么？"
- "Marketing group 在 Q2 立的 campaign 主题是什么？"

测**精确事实记忆**能力。

### 3.2 Memory awareness

模型对自己记忆的元认知：

- "你记得 Charlie 在 leadership group 提到过这件事吗？"
- "我们之前讨论过这个议题吗？"

测**知道自己知道什么**的能力——这是更高阶的认知技能。

### 3.3 User profile understanding

抽象出 user 的长期 profile：

- "Alice 在做决策时倾向于风险厌恶还是风险偏好？"
- "Bob 在过去六个月的工作重点是什么？"

测**跨长时间的抽象提炼**能力。

### 3.4 三维度的互补性

| 维度          | 测什么                | 难点                    |
|---------------|----------------------|-------------------------|
| Fine-grained recall | 精确事实              | 噪声中找细节            |
| Memory awareness    | 元认知                | 知道自己 know/unknow    |
| Profile understanding| 抽象               | 长程跨 session 整合     |

这三个维度对应了记忆系统的三个不同认知功能。一个好的记忆系统需要三方面都强。

## 4. 与 GroupMemBench 的对比

| 维度              | GroupMemBench           | EverMemBench               |
|-------------------|-------------------------|----------------------------|
| 规模              | 50-200 turn / 对话      | 1M+ token / 实例            |
| 群组数            | 单群组                  | 多群组（并行）              |
| Query 类型        | 6 类 adversarial        | 3 维度 holistic             |
| 时间跨度          | 单 session             | 跨多周/多月                 |
| Synthesis 方法    | Graph-grounded         | 三阶段（pipeline+日历+QA）  |
| 难点              | speaker grounding      | 长程 + 跨群组关联           |
| QA 数量           | 30k+                    | 2400                        |
| 数据可重复性      | 高                     | 高                         |

**互补**：

- GroupMemBench 测"在一个群里，speaker 维度处理对不对"
- EverMemBench 测"跨多个群，长时间下，记忆能不能 organic 整合"

**对你的论文**：你应该报这两个 benchmark。一个证明 "你的方法在小群组深 grounding"，另一个证明 "你的方法在大群组长时间还能工作"。这种 dual evidence 让结论更 robust。

## 5. 实验结果（论文报告）

EverMemBench 的实验数据：

| Method        | Recall | Awareness | Profile | Avg  |
|---------------|--------|-----------|---------|------|
| BM25          | 32.1   | 8.5       | 21.3    | 20.6 |
| Mem0          | 41.5   | 12.7      | 28.4    | 27.5 |
| A-MEM         | 43.8   | 15.1      | 30.2    | 29.7 |
| Long-context (Gemini 2.5 Pro) | 48.2 | 22.5 | 35.1 | 35.3 |
| Strongest     | 52.4   | 28.7      | 38.5    | 39.9 |
| Human         | ~80    | ~75       | ~70     | ~75  |

关键发现：

### 5.1 Memory awareness 最差（最强 28.7%）

模型最不擅长 **判断自己记忆的边界**——他们 hallucinate "记得"，或错误地说"不记得"。这是一个非常具体的 weakness，你的方法可以重点优化。

### 5.2 Long-context 模型（Gemini 2.5 Pro）超过记忆系统

注意 Gemini 2.5 Pro 直接把全部 1M+ token 塞进 context window（它能 handle 1M），不用任何 memory system。它的成绩超过 Mem0/A-MEM。

这说明：

- 当 long-context 模型 cap 增大，传统记忆系统的优势缩水
- 但 Long-context 在 awareness 维度仍然不强（22.5%）

**对你的策略意义**：你的 RL 方法应该 **明确不和 Long-context 直接竞争 recall**（那是 cap 之争），而是在 **memory awareness 和 profile understanding** 维度突破。这些是 Long-context 也搞不定的。

### 5.3 Profile understanding 中等

跨长时间抽象 profile 是非常难的——所有方法都在 28-38% 区间。这又是你方法的好优化目标。

## 6. Limitation 与反驳

**Limitation 1：合成（同 GroupMemBench）**。**对你的意义**：用真实公开数据（如 Slack 公开 logs）做 OOD 测试。

**Limitation 2：英文为主，公司场景偏多**。**对你的意义**：做其他场景的 generalization 实验。

**Limitation 3：2400 QA 不算大**。**对你的意义**：注意每个 query 的 evaluation latency，scale 你的 sampling。

## 7. 把它们串起来看：EverMemBench 的整体设计哲学

EverMemBench 的设计哲学是：**真实长程记忆需要超长 scale → 多群组并行交织 → 用真实日历锚点提升真实感 → 用三维度评测覆盖不同认知功能 → 三段质控保证数据质量**。

这是一个 **stress-test benchmark**——它的目的是把记忆系统逼到极限，看在 1M+ token 下还能 work 多少。**和 GroupMemBench 的 "诊断性"互补**。

## 10. 一个具体的实验设计 angle

EverMemBench 的 1M+ token 规模会让你的 RL 训练 cost 爆炸。**实际操作建议**：

### 10.1 训练用 GroupMemBench

GroupMemBench 的小 scale 适合 RL training rollout（每个 trajectory 几千 token 而非百万）。

### 10.2 测试用 EverMemBench

训练好后，在 EverMemBench 上做 **zero-shot / OOD evaluation**——展示你的方法在更大 scale 下的泛化。

### 10.3 类比 Mem-α 的"小训练大测试"叙事

> "We train on GroupMemBench-style data with ~100 turn conversations and evaluate on EverMemBench's 1M+ token instances, demonstrating ~10× length generalization—analogous to Mem-α's 13× generalization from 30k to 400k tokens."

这种叙事是 AAAI 论文的强卖点。

## 12. 一个细节但重要的提醒

EverMemBench 的 Profile understanding 维度需要"跨长时间的抽象提炼"。这是 G-Memory 的 Insight Graph 层和 MOOM 的 Character Branch 都在做的事。

**你的方法在这维度的设计灵感**：

- 借用 G-Memory 的 insight graph 思路 → 跨 session 提炼 speaker insight
- 借用 MOOM 的 Character Branch → keyword-driven 抽取 speaker profile
- 用 RL 学**何时该 insert insight / 何时该 update profile**

这种 EverMemBench-targeted 的方法设计能让你在 profile understanding 维度做出明显提升。
