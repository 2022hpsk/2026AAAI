# SpeakerMem-R1 具体实现细节文档

**用途**：把方法的"输入 / 输出 / 每步目的 / 训练流程"讲到可实现、可答辩级别，并用具体示例说清核心能力与普适性。
**版本**：v1.1（2026-06-02），与 `SpeakerMem-R1-v2-方法spec.md`、`论文草稿-Method章节.md` 对齐。

---

## 0. 一句话核心能力（定调）

> **在多角色（3+ 人 / 群聊）对话中，精准区分"谁说了什么"——把不同说话人的事实、观点、答案分得清清楚楚；回答问题时，准确归属到正确的人、正确的时间、正确的语境，绝不张冠李戴。**

这是本方法要训练出来的**主能力**。它不是某一个垂直场景的小功能，而是一切"多人 + 长期 + 需要记住谁是谁"的系统的**地基能力**——见 §2 的六类难题与跨领域示例。

> 对比：dyadic（一对一）系统会把所有人的发言压成一个 "user"，存储时就丢失了"这是谁的信息"——多人讨论同一话题时，它根本分不清谁是谁。这正是我们要补上的结构性缺口。

## 1. 推理时数据流（用的时候：一进一出）

### 1.1 总输入 / 总输出

- **总输入**：多方对话历史 `C = (u_1, …, u_T)`，每条 `u_t = (说话人 s_k, 文本)`；查询 `q = (谁在问 s_ask, 问关于谁 s_target, 问题文本)`。
- **总输出**：答案 `a`。

### 1.2 内部数据流

```
对话流 C (带说话人标签)
  │  逐轮 turn-by-turn（在线）
  ▼
[A_mem 记忆构建]  每轮: 读当前记忆摘要 + 新发言 → 产出一个记忆动作 → 更新 M
  │
  ▼
记忆 M (5 层, 按说话人分桶)
  │   ←──── 查询 q 到来
  ▼
[A_ret 检索]  按"提问者可见 + 相关"取片段 → context
  │
  ▼
[A_ans 回答]  基于 context + 适配提问者 → 答案 a
```

> **关键事实**：只有**一个模型（一个 policy）**，它换三套 prompt 扮三个角色 A_mem / A_ret / A_ans。训练更新的就是这一个模型（joint multi-agent，借鉴 CoMAM）。

### 1.3 三个角色逐步 I/O

| 步骤 | 输入 | 输出 | 目的 |
|------|------|------|------|
| **① A_mem 写记忆**（每轮跑，在线） | 当前记忆摘要 + 新一条发言 `u_t=(说话人,文本)` + 在场说话人列表 | **一个记忆动作**：`WRITE / UPDATE / DELETE / SUMMARY / PROMOTE / SUPPRESS / NOOP`；WRITE 强制带 `(speaker_id, audience 受众集合, layer 层, content)` | 把"**谁说的、说给谁、什么类型**"正确写进 5 层记忆 |
| **② A_ret 读记忆**（查询时跑） | 查询 `q` + **提问者 s_ask 有权访问**的记忆（`get_accessible`） | 相关记忆片段 `context` | 按**访问控制 + 相关性**取出该用的记忆 |
| **③ A_ans 答题**（查询时跑） | `q` + 检索到的 `context` | 答案 `a` | 基于检索内容、**适配提问者视角**作答 |

### 1.4 动作格式（A_mem 的输出示例）

```
<action>WRITE</action>
<speaker_id>Alice</speaker_id>
<audience>{Alice, Bob, Carol}</audience>
<layer>per_speaker_core</layer>
<content>Alice 在字节跳动做 NLP</content>
```

## 2. 核心能力：在多角色中精准区分"谁说了什么"（六类难题 + 跨领域示例）

下面六类难题，**横跨企业、医疗、教育、客服、决策、社交等多个领域**，共同证明：只要场景里有"多人 + 需要记住谁是谁"，就需要 speaker-grounded 记忆。每类都标注了**难点（现有系统为何失败）**与**我们的创新解法**。这六类也精准对应 SocialMemBench 的五类失败模式 + 隐私维度，直接驱动我们的 5 层记忆设计。

### 2.0 普适性总览

| # | 能力 | 典型应用场景 | 难点（现有系统为何失败） | 我们的机制（创新点） |
|---|------|------------|----------------------|------------------|
| ① | 单流混淆 | 企业协作群、会议纪要、多 agent 协作 | 多人发言压成一条「user」，归属糊成一团 | speaker_id 动作 + per_speaker_core 层 + 按说话人分桶奖励 |
| ② | 跨说话人引用 | 客服 / CRM、合规审计、法律取证 | 丢失"谁说的"(provenance)，或错记成转述者的事 | 同存"来源 speaker_id + 主体 content" + links 关联 |
| ③ | 同类实体不合并 | 医疗多学科会诊、在线教育（一师多生） | 同类实体（"我的患者/我的作业"）大规模错并 | 唯一 s_owner + per-speaker 分层，结构上禁止跨人 merge |
| ④ | 冲突信念按人分 | 决策 / 评审会议、multi-agent 辩论 | 取"最后一条"或合并成单一结论，丢掉分歧 | per-speaker 各记一份信念，UPDATE 不覆盖他人 |
| ⑤ | per-speaker 时序 | 长期陪伴 / 心理咨询、销售 CRM、健康随访 | destructive 覆盖丢历史，问"现在/之前"答不全 | per_speaker_episodic 时间层 + 非破坏 UPDATE + PROMOTE |
| ⑥ | 群体 vs 个体 + 隐私 | 社交 / companion、企业合规、医疗隐私 | 混淆群体规范与个体偏好；无差别泄漏 | group_insight vs per_speaker_core 分层 + audience 访问控制 + R_leak |

> **一句话**：这不是一个"群聊小功能"，而是企业、医疗、教育、客服、决策、社交等所有多人系统的**通用记忆地基**。

---

### 2.1 单流混淆 —— 多人同主题，归属不能混
- **场景**：企业项目群（Slack / Teams / 飞书）· 团队分工
- **对话**：
  - Alice：模型训练我来负责
  - Bob：数据清洗交给我
  - Carol：我盯评测和指标
  - Dave：前端和 demo 归我
- **提问 → 答案**：「谁负责评测？」→ **Carol**；「Bob 做什么？」→ **数据清洗**
- **难点**：现有系统把四人发言压成一条「user」记忆，分工糊成一团——问"谁做评测"时只能猜，或张冠李戴。GroupMemBench 上这类归属错误占比最高。
- **创新解法**：动作空间强制 `speaker_id` + `per_speaker_core` 分层存 + SpeakerLevenshtein 按说话人分桶奖励——每条事实硬锚定到人。
- **适用领域**：企业协作 · 会议纪要 · 项目管理 · 多 agent 协作

### 2.2 跨说话人引用 —— 记住"谁说的"和"说的是谁"
- **场景**：客服 / CRM、合规审计、法律取证（provenance 是硬需求）
- **对话**：
  - Alice：Bob 上次提过，他想从测试转去做开发
  - Dave：哦？他不是一直做测试吗
- **提问 → 答案**：「Bob 想转岗吗？」→ **是（据 Alice 转述）**；「这条信息谁说的？」→ **Alice**
- **难点**：信息**主体是 Bob、来源是 Alice**。现有系统要么把它记成 Alice 的事，要么丢掉来源——合规 / 审计场景必须保留"谁说的"。
- **创新解法**：每条 entry 同时存 `speaker_id`（来源）与 `content`（主体）+ `links` 关联两个说话人；`READ_CROSS` 带访问控制。
- **适用领域**：客服 / CRM · 合规审计 · 法律取证 · 多 agent 信息溯源

### 2.3 同类实体不合并 —— 大群组里，同类实体不能错并
- **场景**：医疗多学科会诊 / 在线教育（一师多生）
- **对话**：
  - 张医生：我这位患者血糖偏高
  - 李医生：我接的是术后感染
  - 王医生：我的患者心率不齐
- **提问 → 答案**：「李医生的患者什么情况？」→ **术后感染**
- **难点**：多人都在谈"我的患者 / 我的症状"，同类实体极易被合并成一团；规模越大越糊。EverMemBench 上 Oracle 多跳归因仅约 26%。
- **创新解法**：唯一 `s_owner` + per-speaker 分层，从结构上禁止跨人 merge；worst-speaker bonus 不放过最难归属的人。
- **适用领域**：医疗会诊 · 在线教育 · 多用户助手 · 大型社群

### 2.4 冲突信念按人分 —— 各持己见，按人记住不同立场
- **场景**：决策 / 评审会议、multi-agent 辩论
- **对话**：
  - Alice：这版我主张上 PyTorch
  - Bob：我坚持 JAX，编译更快
  - Carol：我都行，听大家的
- **提问 → 答案**：「Bob 倾向哪个？」→ **JAX**；「谁主张 PyTorch？」→ **Alice**
- **难点**：同一议题各人观点冲突，现有系统常取"最后一条"或合并成单一结论，丢掉"谁持什么立场"。决策复盘 / 多 agent 协作必须保留分歧。
- **创新解法**：per-speaker 各记一份信念，UPDATE 不覆盖他人；检索时按 speaker 返回各自立场。
- **适用领域**：决策会议 · 评审 / 投票 · multi-agent 系统 · 舆情 / 民调

### 2.5 per-speaker 时序演化 —— 每个人的状态各有时间线
- **场景**：长期陪伴 / 心理咨询、销售 CRM、健康随访
- **对话**：
  - （第 3 天）Alice：我在 A 公司做后端
  - （第 20 天）Alice：我跳槽去 B 公司了，转做算法
- **提问 → 答案**：「Alice 现在在哪？」→ **B 公司做算法**；「她之前呢？」→ **A 公司后端**
- **难点**：每个说话人的事实独立演化。现有系统 UPDATE 多是 destructive 覆盖，丢掉历史；问"现在 / 之前"答不全。
- **创新解法**：`per_speaker_episodic` 时间索引层 + 非破坏性 UPDATE + PROMOTE（反复出现 → 核心），保留可追溯演化。
- **适用领域**：陪伴 / 心理咨询 · 销售 CRM · 健康随访 · 长期个人助理

### 2.6 群体规范 vs 个体偏好 + 受众 / 隐私
- **场景**：社交平台 / companion、企业合规、医疗隐私
- **对话**：
  - 群（共识）：项目统一用 Slack 沟通
  - Alice（私下，受众不含 Carol）：我自己更习惯飞书；另外我在看新机会，先别让 Carol 知道
- **提问 → 答案**：「项目用什么沟通工具？」→ **Slack（群规范）**；Carol 问「Alice 最近状态？」→ **不泄漏跳槽**
- **难点**：群体级 ≠ 个体级，易混；且回答要看"谁在问"。现有系统既会把群规范当个人偏好，也会无差别泄漏隐私。
- **创新解法**：`group_insight`（群规范）与 `per_speaker_core`（个体）分层 + `audience_set` 访问控制 + `R_leak` 泄漏惩罚。
- **适用领域**：社交 / companion · 企业合规（GDPR） · 医疗隐私 · 多租户助手

> **小结**：六类难题、十余个领域，共同指向同一件事——**记忆操作必须理解 who-talked-to-whom-about-what，并把它正确归属、分层、按人检索**。这就是 SpeakerMem-R1 的主能力，也是它"有用、普适、可扩展"的根源。

## 3. 五层说话人索引记忆（每层装什么）

| 层 | 作用域 | 内容类型 | 示例 | 对应失败模式 |
|----|--------|----------|------|-------------|
| `per_speaker_core` | 单说话人 | 持久核心事实 | "Alice 在字节做 NLP" | 单流混淆 |
| `per_speaker_episodic` | 单说话人 | 时间索引情节 | "Alice 上周提到想换工作" | 时序状态覆盖 |
| `per_speaker_profile` | 单说话人 | 风格 / 偏好 | "Bob 习惯用正式英文" | 实体合并（唯一 owner） |
| `group_interaction` | 群组 | 跨说话人事件 / 关系 | "Alice 和 Bob 共事过" | 跨人格知识鸿沟 |
| `group_insight` | 群组 | 高层元知识 | "这个群关注 AI 创业" | 规范-个体混淆 |

- 每条目 `= (entry_id, content, speaker_id 来源, audience_set 受众, layer, turn_created, confidence, links)`。
- `audience_set` 即**显式访问控制**：检索时 `get_accessible(s_ask)` 只返回提问者有权看的条目。

## 4. 训练流程（训练是个啥样子）

### 4.1 训练数据：一条样本长啥样（关键：三个 benchmark 都没训练集 → 全合成）

```
一条样本 = {
  dialogue:       多方对话（带说话人标签）
  memory_gt:      标准记忆（按 speaker 分桶）            ← 给 R_state 当答案
  attribution_gt: 每个 fact 归属谁                        ← 给 R_attr
  privacy_rules:  谁的信息不能给谁                         ← 给 R_leak
  qa_pairs:       [(asker, target, 问题, 标准答案, 依赖记忆片段) × 3]  ← 给 R_task
}
```
来源：GPT-4 合成 200 条（朋友 / 职场 / 兴趣 / 家庭场景）+ LoCoMo dyadic→triadic 扩增。每条须通过质量校验（每说话人 ≥3 facts、含时态变化、含跨说话人引用、QA 必须依赖记忆而非常识）。

### 4.2 奖励函数 = 训练的"打分老师"（定义了"要达到什么目的"）

```
R = 0.5·R_task + 0.8·R_state + 0.3·R_attr + 0.2·R_aud + 0.1·R_compr + 0.1·R_RIF − 0.3·R_leak
     结果对不对   稠密过程(主导)  归属准      受众适配    别膨胀       该忘就忘     跨人泄漏(惩罚)
```

- **R_task**：最终答案 vs 标准答案的 token-level F1（用 F1 不用 binary EM，否则 G=4 时梯度为 0）。
- **R_state（主导，0.8）**：**每一轮、对每个说话人**，比"模型这轮写入的记忆增量 Δ_pred"与"标准增量 Δ_gt"——嵌入余弦相似度 + Hungarian 匹配 + 阈值 τ + 词面保真度，得 `LevF1_s`；再 `平均 + μ·min_s(LevF1_s)（worst-speaker bonus）`。这是**稠密**信号，每轮都告诉模型"这步记忆写得对不对、归属对不对"。
- **R_attr**：说话人归属准确率（哪个 fact 属于哪个说话人）。
- **R_aud**：回答是否适配提问者视角。
- **R_compr / R_RIF**：压缩率（别让记忆无限膨胀）/ 遗忘恰当性。
- **R_leak**：跨说话人信息泄漏惩罚（Alice 私密信息出现在 Carol 可见处则扣分）。

### 4.3 三阶段训练

| 阶段 | 输入 | 过程 | 学到什么 | 目的 / 通过标准 |
|------|------|------|----------|----------------|
| **Stage 1 · SFT 热启动** | 合成对话 + 标注记忆动作 | `speaker-masked 交叉熵`监督 | 会按**说话人归属**输出**合法格式**的记忆动作 | 让模型先"会写"。attribution F1 > 50%（Memory-R2 消融：extraction 不行 → F1 从 49.67 崩到 28.30，所以必须热启动） |
| **Stage 2 · Joint RL（核心）** | 合成对话 batch | SpeakerMem-GRPO（见 §4.4） | 做出**归属对、不混淆、不泄漏、压缩得当**的记忆策略 | 用稠密奖励把策略调好；课程 K=3→5→8 × session 8→16→32 |
| **Stage 3 · 端到端微调** | 三 benchmark 混合(0.5/0.3/0.2) | rollout G=8 + GRPO + 格式惩罚 | construction+retrieval+answer 端到端协同 | 防单 benchmark 过拟合，回填全部指标 |

超参（节选 spec §6）：base = Qwen3-8B（备选 Qwen2.5-7B）；LoRA r=32；Stage1 3 epoch / lr 2e-4；Stage2 1000 step / G_global=4 / G_local=4 / λ=0.3 / β_kl=0.01 / lr 1e-5；Stage3 500 step / G=8 / lr 5e-6；4×A100 共约 20h。

### 4.4 Stage 2 的 RL 循环：一条数据怎么走一遍

```
1. 拿一条对话 → 当前 policy 从头到尾处理（逐轮 A_mem + 末尾答题）= 一条完整轨迹
   重复采样 G=4 次 → 4 条不同轨迹
2. 给每条轨迹用完整奖励 R 打分 → 4 个分数
3. group-relative：高于这组平均分 → 正 advantage；低于 → 负
4. GRPO 更新：抬高高分轨迹里那些记忆动作的概率，压低低分的
5. 【SC-LoGo 局部分支】在轨迹中段取一个 checkpoint，
   强制"相同中间记忆状态 + 相同在场说话人"，再分叉 rollout 4 条，
   同样打分 + group-relative → 局部 loss
   （目的：公平比较，排除"起点不同 / 在场的人不同"导致的不可比）
6. L_total = L_global + λ·L_local + β·KL（防 Echo Trap）→ 更新这一个模型
```

## 5. 易混淆点澄清

1. **一个模型 / 三个角色**：A_mem、A_ret、A_ans 是同一个 policy 换 prompt；训练更新它一个。
2. **A_mem 在线逐轮跑**（对话边来边写）；A_ret / A_ans 在查询时才跑。
3. **一条"轨迹 / episode"** = 处理完整段对话（很多步 A_mem）+ 回答查询。
4. **奖励分两层**：`R_state` 是逐轮稠密反馈，`R_task` 是 episode 末的结果反馈。
5. **所有"标准答案"**（memory_gt / attribution / privacy / QA）都来自合成数据的标注。

---

## 6. 一句话把全流程串起来

> 对话逐轮进来 → 模型每轮决定怎么把"**谁说了什么、说给谁**"写进 5 层记忆 → 查询来了就按"提问者可见 + 相关"检索 → 适配提问者作答。训练时让模型把同一段对话采样跑多遍，用"记忆写得对不对（稠密、按说话人分桶）+ 答得对不对（结果）+ 没混淆没泄漏"打分，GRPO 抬高高分做法，分三阶段（先会写 → 调策略 → 端到端）练出来。**最终得到一个在多角色场景能精准区分"谁说了什么"的通用记忆 agent。**
