# 数据格式 + 评测流程详解（带例子）

这份文档讲清楚三件事：**三个多方记忆 benchmark 的数据长什么样（带真实样例）**、
**各自的官方评测流程**、以及**我们这边统一怎么跑**。看完你应该能完全看懂 `run_baseline.py`。

> 三个 benchmark 都是"多方对话 + 记忆"：一个群里好几个人说话，问题往往绑定"谁问的/谁说的"，
> 考记忆系统能不能分清"谁说了什么、对谁说、什么时候说、后来有没有改口"。这正是本项目
> SpeakerMem-R1 的主战场。

## 0. 三个 benchmark 一览

| | GroupMemBench | SocialMemBench | EverMemBench |
|---|---|---|---|
| 场景 | 企业项目群聊 | 社交群组（朋友/家庭/兴趣…） | 职场长周期协作 |
| 取数 | `setup.sh` 克隆仓库（数据在仓库内） | `get_socialmembench.py`（HF） | `get_evermembench.py`（HF） |
| 规模 | 4 领域×~3万消息，**745 题** | 43 网络/7355 轮/**1031 题** | 1263 对话块/**2400 题**/170 人设 |
| 问题分类 | 6 类 | 9 类 (SR…DM) | 开放式 + 多选 |
| 答案形态 | 短答（日期/实体/拒答） | 长答 + 多选 | 短答/长答 + 多选 |
| 官方判分 | gpt-5 judge | LLM judge（按 condition 跑记忆系统） | OpenRouter：gpt-4.1-mini 答 + gemini judge |
| 我们接了吗 | ✅ run_baseline.py | 🟡 已下数据，loader 待写 | 🟡 已下数据，loader 待写 |

下面逐个展开。所有"例子"都是从真实下载的数据里摘的。

## 1. GroupMemBench

### 1.1 对话文件
路径：`GroupMemBench/data/final/<领域>/synthetic_domain_channels_rolevariants_<领域>.json`
（领域 = Finance / Technology / Healthcare / Manufacturing）

结构：一个大 JSON 字典，**key 是频道名**，value 是该频道按时间排序的消息列表。
**真实样例（一条消息）**：
```json
{
  "msg_node": "Msg_678",
  "content": "We've got the SAR Reporting Requirements phase kicked off ...",
  "author": "User_9",
  "role": "Business Analyst",
  "timestamp": "2025-07-09T01:42:41",
  "reply_to": "Msg_677",                 // 回复哪条 —— 多方对话是"图"不是"链"
  "phase_name": "SAR Reporting Requirements",
  "topic": "Regulatory Reporting Automation",
  "is_noise": false, "is_decision_point": false
}
```
一个领域约 **3 万条**消息、6 个频道、12 个用户（User_1…User_12）。

### 1.2 问题文件
路径：`GroupMemBench/questions/<领域>/<类型>.jsonl`（一行一题）。**真实样例**：
```json
{"id": "multi_hop_1",
 "question": "What is the deadline for Reporting to validate fit against the enterprise-wide ESG policy standard? (YYYY-MM-DD)",
 "answer": "2025-07-18",
 "asking_user_id": "User_7"}            // ★ 谁问的 —— per-asker binding
```
6 类问题（每类专门戳一种能力，括号内是论文里的难点数字）：
- `multi_hop` 多跳：串多条消息才能答
- `knowledge_update` 知识更新：旧信息被新信息覆盖（论文最惨，27.1%）
- `term_ambiguity` 术语歧义：同词不同人含义不同（speaker grounding 的直接考点，37.7%）
- `user_implicit` 用户隐含：答案藏在某人的人设/背景里
- `temporal` 时间推理：先后顺序
- `abstention` 拒答：对话里根本没有 → 正确答案是"无相关信息"

总计 **745 题** = Finance 214 + Technology 213 + Healthcare 145 + Manufacturing 173。

### 1.3 官方评测流程（也是我们 run_baseline.py 做的）
`baselines/{bm25,text_embedding_3_large}/` 提供"检索器"，公共 QA 流程在 `baselines/rag_common/eval_lib.py`：
```
读对话 → 建索引 → 对每题：检索 top-k 段落 → gpt-5 QA-agent 作答 → gpt-5 judge 判对错 → 按 6 类统计准确率
```
公开版本只放了 **BM25 + text-embedding-3-large** 两个 baseline；Mem0/A-MEM 等没放出来（我们自己重建）。

## 2. SocialMemBench

来源：HF `anon4data/socialmembench`，4 个配置靠 `network_id` 关联（已下载到 `SocialMemBench/`）。

### 2.1 四个配置
- **`networks.jsonl`（43）**：群组。字段 `network_id, group_name, group_type(5类), group_size,
  tier(small/medium/large), edges_json(关系图), group_norms_json(群规)`。
- **`personas.jsonl`（430）**：人设。字段 `persona_id, display_name, age, gender, occupation,
  big_five_json(大五人格), communication_profile_json, preferences_json,
  preference_history_json(偏好随时间变化), speaking_quirks_json`。
- **`conversations.jsonl`（7355，348 session）**：长表，一行一句。**真实样例**：
```json
{"network_id": "grp_0d1e2f3a", "session_id": "grp_0d1e2f3a_s01",
 "session_topic": "Quarterly drinks planning; Leon's startup announcement",
 "turn_id": "grp_0d1e2f3a_s01_t000", "speaker_display_name": "Josh",
 "timestamp": "2025-10-06T12:08:00",
 "message": "Quarterly drinks! Last Thursday of the month? I've found a new place in Soho 😁",
 "reply_to_turn_id": ""}
```
- **`qa.jsonl`（1031）**：问答。**真实样例（长答型）**：
```json
{"qa_id": "Q1_c0s3c1", "network_id": "grp_0d1e2f3a", "query_type": "Q1",
 "difficulty": "medium", "answer_format": "long_form",
 "question": "What does Josh's behavior in the group chat suggest about how he handles questions about his own career?",
 "answer": "Josh consistently deflects questions about his career with self-deprecating humour and then redirects ...",
 "evidence_anchors_json": "[{\"session_index\":3,\"turn_id\":\"...s03_t002\",\"speaker_display_name\":\"Josh\",\"message_excerpt\":\"...\"}]"}
```
还有多选题（`options_json` 非空、`correct_option` 是字母）。难度分布：medium 727 / hard 215 / easy 89。

### 2.2 九类问题（HF 的 query_type 是 Q1–Q9，官方代码/README 用下面的代号，一一对应）
| 代号 | 名称 | 考什么 |
|---|---|---|
| SR | 单人召回 | 某个人隐含的偏好 |
| GD | 群体决策 | 决策 + 谁反对 |
| MA | 多人聚合 | 同一话题下所有人的立场 |
| AP | 归属探针 | 两个易混的人，谁说的要分对 |
| TM | 心理揣测(ToM) | A 透露了 B 的什么偏好 |
| NI | 群规 vs 个人 | 某条群规是否对每个人都适用 |
| RE | 关系边 | 聊天中暴露的关系史 |
| TS | 时间漂移 | 偏好如何跨 session 改变 |
| DM | 离群成员 | 已退群成员最后已知的偏好 |

### 2.3 官方评测流程
```bash
pip install -e .
python -m eval.cli --condition graphiti --network grp_a0b1c2d3
```
`--condition` 就是被测的记忆系统（如 graphiti、mem0…）。质量：blind-critic 均值 0.952，每题人工复核。
> 形态和 GroupMemBench 不同（长表 + 多选/长答 + 证据锚点），所以要单独写 loader：
> 把 conversations 按 network_id+session 组装成"对话"，qa 转成统一问答格式（多选要把 options 拼进 prompt）。

## 3. EverMemBench

来源：代码库 `EverMemBench/`（GitHub EverMind-AI/EverMemBench）+ 数据 HF `EverMemBench-Dynamic`
（已下载到 `EverMemBench-Dynamic/`）。场景：职场长周期协作，跨约 250 天，围绕"碳排放会计 + 资产管理平台"项目。

### 3.1 三个配置
- **`dialogues.jsonl`（1263）**：按 `topic_id`(01–05) + `date` 组织，`dialogues` 是字典 `{Group 1/2/3: [消息…]}`。
  **真实样例（一条消息）**：
```json
{"speaker": "Weihua Zhang", "time": "2025-01-09 09:32:15",
 "dialogue": "Good morning everyone. Today, we officially launch the Carbon ... project",
 "message_index": "1"}
```
- **`qars.jsonl`（2400）**：问答三元组 Q/A/R。**真实样例（开放式 + 多选各一）**：
```json
// 开放式
{"id": "F_SH_Top01_001", "topic_id": "01",
 "Q": "After SQL optimization of the data submission API, what was the peak CPU usage during the final 300 concurrent ...?",
 "A": "65%",
 "R": [{"date": "2025-10-20 00:00:00", "group": "Group 3", "message_index": "4"}, ...],  // 证据锚点
 "options": null}
// 多选
{"id": "MA_C_Top01_001", "topic_id": "01",
 "Q": "Legal just informed us that our carbon emissions ...",
 "A": "D",                                   // 多选答案是字母
 "R": [{"date":"2025-01-21 ...","group":"Group 1","message_index":"1-6"}, ...],
 "options": {"A": "...", "B": "...", "C": "...", "D": "..."}}
```
  注意 R 里的 `message_index` 可能是 `"4"` 或区间 `"2-3, 6-7"`（多条证据）。
- **`profiles.jsonl`（170）**：员工档案。字段 `Name, ID, Gender, Age, Education, Major, Dept, Title,
  Rank, Tenure, Skills_List, Communication_Profile, Big_Five_Profile, Interests, Marital_Status`。

### 3.2 三层评测维度
factual recall（事实召回）/ applied reasoning（应用推理）/ personalized generalization（个性化泛化）。
qars 的 id 前缀（如 `F_SH_`、`MA_C_`）编码了维度与类型。

### 3.3 官方评测流程（四阶段：Add → Search → Answer → Evaluate）
代码在 `EverMemBench/eval/`，CLI 是 `eval/cli.py`：
```bash
# 1) 入库（把对话灌进记忆系统）
python -m eval.cli --dataset dataset/004/dialogue.json --system memos --user-id 004 --stages add
# 2) 检索 + 作答 + 判分
python -m eval.cli --dataset dataset/004/dialogue.json --qa dataset/004/qa_004.json \
    --system memos --user-id 004 --stages search answer evaluate --top-k 10
# 也支持 --system llm：把整段对话当 context 直接喂 LLM（long-context 上界对照）
```
- 被测记忆系统(`--system`)：**Memos / Mem0 / Memobase / EverMemOS / Zep**，外加 `llm`(long-context)。
- 默认模型（`eval/config/pipeline.yaml`，走 OpenRouter）：作答 `openai/gpt-4.1-mini`，判分 `google/gemini-3-flash-preview`。
- 数据模型（`eval/src/core/data_models.py`）：`GroupChatMessage(speaker/content/timestamp/group/date)`、
  `QAItem(question_type ∈ {multiple_choice, open_ended})`、结果 `accuracy_by_type`。
> 注意：仓库 CLI 吃的是 `dataset/{batch}/dialogue.json + qa_{batch}.json` 这种"按 batch 的文件"，
> 与 HF 的 `EverMemBench-Dynamic`（dialogues/qars/profiles 三表）形态不同，接入时要做一次格式转换。

## 4. 我们这边统一怎么跑（以 BM25 为例）

`run_baseline.py` 对每个(领域/网络, 问题类型)做的事：
```
第1步 读对话 → 排成消息列表（GroupMemBench：按 频道→时间→msg_node）
第2步 建索引（BM25：每条消息正文切词；元数据只在展示给 LLM 时拼上，不进索引）
第3步 每题：检索 query = "asker + 问题"，取分数最高 top-k=10 条 → 格式化成段落
第4步 QA-agent（1 次 LLM）：喂【问题 + top-k 段落】，要求"先思考，再 Final: <答案>"
第5步 judge（1 次 LLM）：喂【问题 + 标准答案 + agent答案】，判 "Final: Correct/Incorrect"
第6步 统计：judge=Correct 的比例 → 按问题类型 + 总体平均
```
- 第1~3步**不用 LLM**；第4~5步每题 **2 次 LLM 调用**。一轮完整 GroupMemBench ≈ 745×2 ≈ **1490 次调用**。
- **Mem0 与 BM25 唯一区别**在第2、3步：Mem0 先用 LLM 把对话"抽取/概括"成记忆（入库就烧 LLM），
  检索回来的是"记忆文本"而非原始消息；第4~6步两者一致（保证可比）。所以 `run_mem0.py` 更贵、单独写。
- LLM 现走 **DeepSeek（deepseek-v4-flash，OpenAI 兼容）**，key 在 `.env`。
  **加 `--retrieval-only` 就只做第1~3步、完全不调 LLM**（现在就能验证检索链路，零费用）。
