## 2. 评测流程（统一，以 BM25 为例）
`读对话 → 建索引 → 每题检索 top-k 段落 → QA-agent 作答(先思考后 Final:) → judge 判对错 → 按类别统计准确率`
- 检索阶段不调 LLM；问答阶段每题 2 次 LLM（agent + judge）。一轮 GroupMemBench 全量 ≈ 745×2 ≈ 1490 次调用。
- 适配：`llm_clients.py` 把上游 OpenAI 形态调用路由到 DeepSeek；`eval_patches.py` 抬高 token 预算 + pin temperature=0。

## 3. BM25 baseline —— 已真实出数 ✅
- **GroupMemBench / Finance / multi_hop / 20 题 = 35.0%**，与论文 BM25 multi_hop ≈ 38.2% **量级吻合** → 流程忠实。
- 跑这一笔顺带修了两个"上游为 gpt-5 调参、换 DeepSeek 不合适"的坑：
  1. judge/agent token 太小 → DeepSeek 啰嗦 reasoning 把 `Final:` 行截断 → 误判 Unclear（Unclear 5→1）。
  2. temperature=0.2 不可复现 → pin 为 0。

## 4. Mem0 baseline —— 管线打通 + 性能优化 🟡→✅(管线)
全链路打通：消息经 DeepSeek 抽取入库 → 本地 chroma → 检索 → QA → judge。首测 0%/24 分钟，深挖后定位并修复：

| 问题 | 根因 | 修复 | 效果 |
|---|---|---|---|
| 检索/embedding 极慢 | 192 核致 torch 线程超额订阅 | 限 8 线程 | 检索 14.5s → **0.02s**（↓700×）|
| 重复入库浪费 token | get_all 签名用错被吞异常 | 持久化 chroma + 跳过已入库 | 入库一次可复用 |
| 抽取 0 条事实(0%) | deepseek-v4-flash 是推理模型，mem0 抽取 token 不够→JSON 被截断 | max_tokens 调到 8000 | 实测 4000 即可正常抽出事实（待实跑验证）|
| 遥测卡顿 | mem0/chromadb posthog 网络超时 | 关闭遥测 | 次要提速 |

- **结构性成本**：入库每批仍要 1 次 DeepSeek 抽取（~20s/批）；全量 30k 条需专门规划，但**入库一次即可持久化复用**。
