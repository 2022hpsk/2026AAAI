LOCOMO（Maharana et al., ACL 2024, arXiv 2402.17753）是 **dyadic 长期对话记忆**的事实标准 benchmark。在 2024 年发表后，几乎所有后续记忆系统论文（Mem0、A-MEM、Memory-R1、AgeMem、Mem-α）都在 LOCOMO 上报数。**作为 dyadic 对照，你的多角色论文必须报 LOCOMO，证明你的方法在 dyadic 不退化。**

## 1. "50 个高质量超长对话" —— 数据规模

LOCOMO 的核心数据：

- **50 个对话**（不算大但精）
- **每个平均 300 turn**
- **每个约 9K token**
- **每个至多 35 session**
- 基于 persona + 时序事件图生成
- 含图像（multimodal task）

### 1.1 为什么 50 个就够

50 个对话听起来不大，但每个都精心设计：

- 300 turn 的 dialogue 比 PersonaChat（10 turn）长 30 倍
- 35 session 跨数月，模拟真实长期交往
- persona 经过 LLM + human verification 保证一致性
- 时序事件图保证 longitudinal consistency

这种 "精而非多" 的数据让 LOCOMO 成为 **质量基准**——衡量记忆系统在精心控制条件下的能力。

### 1.2 评测时的 query 数量

每个对话约 200 个 problem（问题），50 对话 × 200 = **10,000 QA**。这个数量足够 robust 评测。

## 2. "Persona + 时序事件图生成" —— 数据 pipeline

LOCOMO 的数据构造 pipeline 是 multi-step：

### Step 1: Persona generation

```python
persona = {
    "name": ...,
    "age": ...,
    "occupation": ...,
    "personality_traits": [...],
    "interests": [...],
    "goals": [...],
    "background": "..."  # 详细背景故事
}
```

每个 persona 有 ~20 个属性，覆盖多个维度。

### Step 2: Temporal event graph

```python
# 设计这个 persona 在 session 期间会经历的事件
events = [
    {"date": "Week 1", "event": "started new job", "impact": "career change"},
    {"date": "Week 2", "event": "met colleague Alice", "impact": "new friend"},
    {"date": "Week 4", "event": "promoted", "impact": "responsibility increase"},
    ...
]

# 事件之间有依赖关系（图）
edges = [
    ("started new job", "met colleague Alice"),  # 因果
    ("met colleague Alice", "promoted"),         # Alice 帮助升职
    ...
]
```

### Step 3: Session generation

每个 session 是一次对话。Session 之间间隔几天到几周。每个 session 生成时，LLM 会：

1. Refer 到 persona 属性
2. Refer 到时序事件图
3. 生成 user 和 AI 之间自然的对话

这种生成保证了 **跨 session 的 longitudinal consistency**。

### Step 4: 图像注入

LOCOMO 引入图像 attachment——某些 turn 会包含图片（比如 user 分享了一张照片）。这测多模态记忆能力。

### Step 5: Question generation

每个 session 后生成 evaluation questions，覆盖五类（见下）。

## 3. "三任务：QA、Event Summarization、Multimodal Dialog Generation"

LOCOMO 是 multi-task benchmark：

### 3.1 QA（主任务）

五类 QA：

1. **Single-hop**：问题答案在某 single turn 里
   - 例："Alice 上次提到她在哪工作？"

2. **Multi-hop**：跨多 turn 推理
   - 例："Alice 升职前的职位是什么？"（需要找升职 turn + 之前职位 turn）

3. **Temporal**：时间推理
   - 例："Alice 是几月份升职的？"

4. **Open-domain**：开放生成（不是 yes/no）
   - 例："总结 Alice 过去三个月的工作变化"

5. **Adversarial**：误导性问题
   - 例："Alice 在 Google 工作了几年？"（陷阱：Alice 没在 Google 工作过，正确答案应该是拒答）

### 3.2 Event Summarization

把一段对话总结成 events list。测**信息提炼能力**。

### 3.3 Multimodal Dialog Generation

生成包含图像引用的对话。测**多模态记忆**。

## 4. "F1 + LLM-as-judge" —— 评测指标

LOCOMO 早期主要用 F1（partial-match），后来增加 LLM-as-judge。

### 4.1 F1 (partial match)

```python
def compute_f1(prediction, gold):
    pred_tokens = set(normalize(prediction).split())
    gold_tokens = set(normalize(gold).split())
    if not pred_tokens or not gold_tokens:
        return 0
    
    common = pred_tokens & gold_tokens
    precision = len(common) / len(pred_tokens)
    recall = len(common) / len(gold_tokens)
    f1 = 2 * precision * recall / (precision + recall) if (precision + recall) else 0
    return f1
```

这种 token-level overlap 简单但能反映大致正确性。

### 4.2 LLM-as-judge

```python
def llm_judge(prediction, gold, question):
    judgment = GPT4.judge(
        f"Question: {question}\n"
        f"Gold: {gold}\n"
        f"Prediction: {prediction}\n"
        f"Is the prediction semantically correct? 0/1"
    )
    return judgment
```

LLM-as-judge 比 F1 更宽容（允许 paraphrase），但有 LLM 偏见。

### 4.3 双指标的价值

报 F1 + LLM-judge 双指标更 robust——F1 可能 punish paraphrase，LLM-judge 可能 over-praise。两者一致是 strongest 证据。

## 5. "Memory-R1/Mem0/AgeMem 都在此报数" —— Comparability

LOCOMO 成为标准的重要原因是 **comparable baselines**：

| Method        | F1   | BLEU-1 | LLM-as-Judge |
|---------------|------|--------|--------------|
| OpenAI memory | 0.32 | 0.21   | 0.45         |
| Mem0          | 0.40 | 0.28   | 0.62         |
| A-MEM         | 0.38 | 0.27   | 0.60         |
| MemoryBank    | 0.35 | 0.25   | 0.55         |
| Memory-R1     | 0.59 | 0.47   | 0.85         |
| Mem-α         | 0.62 | 0.50   | 0.87         |
| AgeMem        | 0.65 | 0.52   | 0.89         |

（注：数字基于论文报告趋势估算）

你的论文必须在这个表格里出现，并显示你的方法**至少接近** AgeMem/Mem-α 水平——否则审稿人会质疑 dyadic 性能。

## 6. 你的论文里 LOCOMO 的角色定位

注意：**LOCOMO 不是你的主战场**，但它是**必须的对照**。具体定位：

### 6.1 角色 1：dyadic compatibility check

证明你的多角色方法在 dyadic 场景下不退化。这是 reviewer 必问的——"加了 speaker 维度，dyadic 还能 work 吗？"

### 6.2 角色 2：方法横向对比

让 reader 把你的方法和 Mem0/AgeMem 等放在同一坐标系。LOCOMO 是这个坐标系的横轴。

### 6.3 不是角色：主要 contribution 的 evidence

你的主 contribution 应该体现在 GroupMemBench/EverMemBench/SocialMemBench 上。LOCOMO 上只需要 "不显著差于 SOTA dyadic 方法"。

### 7.2 期望性能

你的方法在 LOCOMO 上不需要超过 AgeMem 太多——0.65±0.02 的 F1 就足够（接近 AgeMem 的 0.65）。重点是**不能掉**——如果你的多角色方法在 dyadic 上反而下降，方法 design 有问题。

## 8. Limitation 与反驳

**Limitation 1：dyadic only**。LOCOMO 不测多方场景。

**Limitation 2：合成（同其他）**。

**Limitation 3：英文为主**。

**Limitation 4：partial-match metric 偏 strict**。LLM-judge 缓解但不彻底。

## 9. 把它们串起来看：LOCOMO 的整体设计哲学

LOCOMO 的设计哲学是：**质量先于数量 → 精心 design 跨 session longitudinal 一致性 → 多任务覆盖（QA / Summary / Multimodal）→ 双指标（F1 + LLM-judge）→ 配套强 baseline 形成可比 ecosystem**。

这是 ML benchmark 的成功范式——**LOCOMO 的成功**说明 carefully crafted 50 个高质量样本比 million-scale 但低质量样本更有研究价值。

## 12. 一个 practical 提醒

LOCOMO 的 GitHub 仓库提供了：

- 数据下载
- evaluation script
- baseline code（基础 LLM、RAG 等）

你应该 **直接用官方 evaluation script** 报数。**不要自己实现 F1**——可能会有 normalization 差异，让你的数字看起来比实际好/差。这种"使用官方 script"的细节在 reviewer 看来是 trust signal。

## 13. 一个 timing 的 historical 视角

LOCOMO 在 2024 年初发表，到 2026 年（你的 AAAI-27 timeframe）已经 2 年。这意味着：

- 几乎所有 dyadic 方法都报过 LOCOMO 数字
- 数字 saturate 较高（F1 接近 0.65 是当前最强）
- **要在 LOCOMO 上 +5 个点已经很难**

**对你的意义**：不要把 LOCOMO 当作 main innovation 战场。在它上面"持平"已经是好结果。把精力投入到 multi-party benchmarks 的提升上——那里有 +30 个点的提升空间。

这种 **strategic deprioritization** of saturated benchmark 是 ML 论文的成熟思考方式。审稿人会理解。
