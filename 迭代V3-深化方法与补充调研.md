## Part 1：V3 新发现论文（2026 年 1-5 月未在原调研覆盖）

### 1.1 [关键竞品] MemBuilder（arxiv 2601.05488，2026/1 月）

**全名**：MemBuilder: Reinforcing LLMs for Long-Term Memory Construction via Attributed Dense Rewards  
**作者单位**：Sun Yat-sen University + Tencent  

**核心创新**：
1. **Session-level 中间奖励**：每个 session 结束自动合成 QA，给即时 feedback（解决稀疏奖励）
2. **贡献感知梯度加权**：记忆操作的梯度权重 ∝ 该操作产生的片段被检索的次数

**四维记忆架构**：Core + Episodic + Semantic + Procedural（与 MemFail 分类高度吻合）

**效果**：4B 模型在 LoCoMo/LongMemEval 上超过 Memory-R1 和多数闭源模型

**对我们的冲击**：
- MemBuilder 是 dyadic，但技术路线（dense reward + multi-dim memory）与我们非常接近
- **必须作为强 baseline** 复现到多方 benchmark 上
- **可借鉴**：贡献感知梯度加权 → 扩展为 speaker-aware 梯度加权
- **可扩展**：四维记忆 → per-speaker 四维记忆（5 × 4 = 20 维矩阵）

**写作注意**：审稿人可能 compare MemBuilder + 多方 benchmark，我们需要清晰说明为什么 MemBuilder 的单流架构在多方场景下失败（speaker confusion 问题）。

### 1.2 [方法启发] DualMem（arxiv 2605.25693，2026/5 月）

**全名**：From Facts to Insights: A Persona-Driven Dual Memory Framework and Dataset for Role-Playing Agents

**核心创新**：
- 双流记忆：Factual（客观事实）+ Insight（persona-conditioned 解读）
- RoleMemo 数据集：4 类推理任务（Factual QA / Persona-Conditional / Implicit Preference / Temporal Consistency）
- SFT + RL 训练，4B 超 zero-shot DeepSeek-V3.2

**对我们的冲击**：
1. DualMem 的 "facts → insights" 双流正好对应我们 SpeakerMem-R1 的 episodic → insight PROMOTE 操作
2. **关键 delta**：DualMem 是单角色（已知 persona），我们是多 speaker（未知、动态 speaker profile）
3. 可以说：SpeakerMem-R1 = DualMem 扩展到多方场景 + RL 训练 speaker-profile 推断
4. 借鉴 RoleMemo 的任务类型，设计 GroupRoleMemo 数据集

### 1.3 [分析工具] MemFail（arxiv 2605.26667，2026/5 月）

**全名**：MemFail: Stress-Testing Failure Modes of LLM Memory Systems

**五大失败模式**：
1. Conditional-Facts：保留条件事实时失败
2. Persona-Retrieval：被误导性问题干扰时检索失败
3. Long-Hop：长链推理时失败
4. Coexisting-Facts：保留多条共存事实时失败
5. Conditional-Facts Hard：隐晦条件保留失败

**被测系统**：Mem0、A-MEM、SimpleMem、StructMem（均为 dyadic）

**对我们的冲击**：
- 在多方场景下，5 个失败模式全部放大（speaker identity 增加额外复杂度）
- 可用于 motivation：quantify failure rates of existing systems in multi-party settings
- 可作为消融测试套件：MemFail-MP（多方扩展版）

### 1.4 [Safety 视角] 记忆安全三论文（2026 年）

**a. Remembering More, Risking More（2605.17830）**
- 时间记忆污染：跨 session 渐进积累有害记忆，单次安全过滤器无法检测
- Trigger-Probe Protocol：用固定探针评估记忆快照的安全性
- **对我们**：多方场景下，N 个 speaker 中 1 个攻击者可污染整个群体记忆

**b. SSGM（2603.11768）**
- 稳定性约束 + 安全性约束双重记忆治理
- Speaker 信任评估：基于历史一致性的动态信任建模
- **对我们**：speaker-trust 层是我们隐私防护的理论基础

**c. Unveiling Privacy Risks（2502.13172）**
- 三类泄露：直接泄露、推断泄露、跨用户传播
- 多方场景特有：Speaker Confusion、Audience Confusion、Group Consensus Override

**对 SpeakerMem-R1 的价值**：
- 为我们的隐私层提供理论支撑
- 开辟了"Safety & Privacy"角度的独特 motivation
- 无论哪篇论文都没有针对多方场景的解决方案 → 我们的 contribution point

### 2.2 v3 完整记忆数据结构定义

```python
@dataclass
class SpeakerMemorySlot:
    """单个 speaker 的四维记忆槽"""
    speaker_id: str
    
    # Dim 1: Core（核心身份事实）
    core: list[MemoryFragment]      # 不易变、高置信的身份属性
    
    # Dim 2: Episodic（具体事件）
    episodic: list[MemoryFragment]  # 时间戳 + 具体描述
    
    # Dim 3: Semantic（抽象知识/偏好）
    semantic: list[MemoryFragment]  # 推断出的偏好、知识、态度
    
    # Dim 4: Insight（persona-conditioned 洞察，DualMem 风格）
    insight: list[MemoryFragment]   # 由 episodic PROMOTE 来
    
    # 隐私属性
    trust_score: float = 1.0        # SSGM speaker trust
    quarantine: list[MemoryFragment] = []  # 可疑信息隔离区

@dataclass  
class GroupMemorySlot:
    """群体层面的记忆"""
    interaction_patterns: list[str]  # 谁和谁在什么话题上互动
    group_insight: list[str]         # 群体动态洞察
    consensus: dict[str, str]        # 话题 → 群体共识
    dissent: dict[str, list[str]]    # 话题 → 持异见的 speaker list

@dataclass
class MemoryFragment:
    content: str
    speaker_id: str                  # 信息来源 speaker
    audience_set: frozenset[str]     # 该信息对哪些 speaker 可见
    confidence: float                # 置信度（基于 corroboration）
    timestamp: int                   # session index
    layer: str                       # 'core'/'episodic'/'semantic'/'insight'
    retrieval_count: int = 0         # 被检索次数（用于梯度加权）
```

### 2.4 v3 完整奖励函数

```python
def reward_v3(trajectory, ground_truth, speakers):
    """
    SpeakerMem-R1 v3 完整奖励函数
    """
    
    # 1. 任务奖励（最终回答质量）
    R_task = token_level_F1(
        trajectory.final_answer, 
        ground_truth.answer
    )  # 用 token-F1 而非 binary EM（Curriculum Study 建议）
    
    # 2. Speaker 归因奖励（核心新贡献）
    R_attr = speaker_attribution_F1(
        trajectory.memory_state, 
        ground_truth.speaker_memory
    )
    # 衡量：Alice 的信息是否存在 Alice 的桶里，而非混入 Bob
    
    # 3. Audience 适配奖励
    R_aud = audience_adaptation_score(
        trajectory.responses, 
        ground_truth.audience_profiles
    )
    
    # 4. DeltaMem-style speaker-aware Levenshtein（稠密过程奖励）
    R_state = sum(
        speaker_levenshtein_F1(
            trajectory.memory[s], 
            ground_truth.memory[s]
        ) 
        for s in speakers
    ) / len(speakers)
    # 每个 speaker 的记忆状态差都要单独计算
    
    # 5. 压缩奖励（Mem-α 风格）
    R_compr = compression_efficiency(trajectory.memory)
    
    # 6. 适当遗忘奖励（MOOM 风格）
    R_RIF = forgetting_appropriateness(
        trajectory.suppress_ops, 
        ground_truth
    )
    
    # 7. 隐私/安全奖励（v3 新增）
    R_privacy = (
        - 2.0 * cross_speaker_leakage_rate(trajectory)
        - 1.5 * audience_violation_rate(trajectory)
        + 0.5 * quarantine_precision(trajectory)
    )
    
    # 8. 格式奖励（保证输出合法 JSON）
    R_format = format_compliance(trajectory.actions)
    
    # 加权组合（参考 DeltaMem 的权重设计：state reward 给大权重）
    return (
        1.0 * R_task      # 任务质量（主导）
        + 0.5 * R_attr    # speaker 归因（次要主导）
        + 0.6 * R_state   # 记忆状态 Levenshtein（稠密信号）
        + 0.3 * R_aud     # audience 适配
        + 0.2 * R_compr   # 压缩
        + 0.1 * R_RIF     # 遗忘
        + 0.3 * R_privacy # 隐私（新增）
        + 0.1 * R_format  # 格式
    )
```

### 2.5 v3 Speaker-Aware Contribution Gradient Weighting

在 GRPO 训练时，对每个记忆操作 aᵢ 的梯度加权：

```python
def compute_gradient_weight(action, trajectory, speaker_id):
    """
    MemBuilder 贡献感知梯度加权的 speaker-aware 版本
    """
    # 基础：该操作生成的记忆片段被检索的频率
    base_weight = action.memory_fragment.retrieval_count / total_retrievals
    
    # Speaker-aware 修正：
    # 该操作的 speaker attribution 是否正确
    correct_attribution = (
        1.0 if action.speaker_id == ground_truth.true_speaker(action.content)
        else 0.2  # 归因错误的操作降权
    )
    
    # 隐私合规修正：
    # Quarantine 中的操作获得轻微奖励（正确的谨慎态度）
    privacy_weight = 1.0 if action.action_type != 'QUARANTINE' else 1.2
    
    return base_weight * correct_attribution * privacy_weight

# 在 GRPO 更新时：
for action in trajectory.actions:
    w = compute_gradient_weight(action, trajectory, action.speaker_id)
    gradient_update += w * policy_gradient(action)
```

### 2.6 v3 训练流程（三阶段 + LoGo + Joint）

```python
def train_speakermem_r1_v3():
    """
    三阶段训练，融合：
    - AgeMem step-wise GRPO
    - Memory-R2 LoGo local rerollout（speaker-conditioned）
    - CoMAM joint multi-agent training
    - MemBuilder contribution-aware gradient weighting
    - Curriculum Study mixed curriculum
    """
    
    # === Stage 1: Speaker-Aware SFT 热启动 ===
    # 在标注好 speaker 归因的对话数据上做 role-masked SFT
    # 数据：GroupMemBench 训练集 + 自合成多方对话（按 Curriculum Study 建议 150+ 样本）
    speaker_role_masked_sft(
        data=mixed_curriculum,
        roles=['memory_agent', 'per_speaker_roles'],
        loss='masked_CE'
    )
    
    # === Stage 2: Joint RL on Memory Construction ===
    # 同时训 construction + retrieval + write_policy（CoMAM 风格 joint）
    # 使用 speaker-aware LoGo-GRPO（Memory-R2 风格的 local rerollout）
    for batch in training_data:
        # 全局 rollout（端到端）
        global_trajectories = rollout(G=4)  # G=4 per Curriculum Study
        global_reward = reward_v3(global_trajectories)
        
        # 局部 rerollout（speaker-conditioned）
        intermediate_state = sample_checkpoint(global_trajectories)
        for speaker_partition in stratified_speaker_sets(batch):
            local_rollouts = rerollout(intermediate_state, speaker_partition, G=4)
            local_reward = reward_v3(local_rollouts)
            
            # LoGo 组合
            loss = GRPO_loss(global_trajectories, global_reward) \
                 + λ * GRPO_loss(local_rollouts, local_reward)
        
        # Speaker-aware 梯度加权
        weighted_update(loss, gradient_weights=compute_all_weights(batch))
    
    # === Stage 3: End-to-End Full Task RL ===
    # 在三个多方 benchmark 的训练集上做完整 RL
    full_rl_stage(
        algorithm='MultiContextGRPO',  # MemSearcher 风格
        reward=reward_v3,
        eval_benchmarks=[GroupMemBench, EverMemBench, SocialMemBench],
        kl_reg=True,  # 防 Echo Trap（WebAgent-R1 教训）
    )
```

### Idea 9：Speaker-Counterfactual Credit Assignment

**灵感来源**：LEMON（2605.14483）的 localized counterfactual RL

**核心思路**：在多方场景中，credit assignment 的关键问题是"如果我把这个 fact 归因给 speaker A 而不是 speaker B，结果会不会更好？"

**具体方法**：
```python
def speaker_counterfactual_reward(action, trajectory):
    """
    反事实奖励：如果换一个 speaker attribution，结果更好/更差？
    """
    # 原始奖励
    original_reward = reward_v3(trajectory)
    
    # 反事实：把 action 的 speaker_id 换成其他 speaker
    counterfactual_rewards = []
    for alt_speaker in speakers:
        if alt_speaker == action.speaker_id:
            continue
        alt_trajectory = apply_counterfactual(trajectory, action, alt_speaker)
        counterfactual_rewards.append(reward_v3(alt_trajectory))
    
    # 如果原始归因 > 所有反事实：正面强化
    # 如果有反事实更好：负面惩罚
    return original_reward - max(counterfactual_rewards)
```

**优势**：理论上更精确的 credit assignment，避免"模糊归因"的问题  
**劣势**：计算开销 = N × original（N 个 speaker，多 N 倍推理）

**建议**：在小规模（N=3-4 speaker）ablation 中测试，若有效则作为额外贡献。

### Idea 10：Privacy-Preserving Multi-Party Memory（独立方向）

**背景**：记忆安全三论文揭示了多方场景的隐私风险，但没有解决方案。

**我们可以做**：

2. 评测：设计攻击场景（一个 speaker 试图通过群聊污染其他 speaker 的 memory），测试防护效果
3. 独立投稿可能性：AAAI Safety Track / ACL Findings / USENIX Security

**可行性**：中（隐私评测 benchmark 需要设计）  
**价值**：高（安全+AI 是高关注度方向）  
**建议**：作为 SpeakerMem-R1 主论文的 Section 5（Safety Analysis），先验证概念，后续独立投稿。

### 6.3 实验设计 v3

**主实验**：

6. 去掉 multi-dim memory → single stream
