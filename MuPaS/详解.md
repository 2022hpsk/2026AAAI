MuPaS（Multi-Party Supervised Fine-tuning，arXiv 2412.05342，2024 年 12 月）是 **多方对话生成** 这条线上的关键基础工作。它本身不做记忆，但是**多角色场景下"如何让 LLM 在多说话人模型中正确建模 speaker 维度"的标准训练技术**。这套技术是你做 speaker-grounded 记忆策略时**必然要用到的基础设施**。我把它逐层拆开。

## 1. "多方对话" —— 与 dyadic 的根本不同

先把"多方对话（Multi-Party Dialogue, MPD）"和"dyadic 对话"的差异讲清楚。这关系到你为什么 dyadic 工作的技术不能直接搬到多方。

### 1.1 Dyadic 对话

两个参与者轮流发言：

```
User: ...
AI:   ...
User: ...
AI:   ...
```

每个 turn 的 speaker 是确定的（无需建模），addressee 也是确定的（永远是对方），上下文是线性的。

### 1.2 Multi-Party 对话

多个参与者，发言顺序不固定，addressee 不固定：

```
Alice → Bob: "Did you finish the report?"
Bob → Alice: "Almost, just need Charlie's input."
Charlie → Bob: "I'll send it tonight."
Alice → Charlie: "Thanks, deadline is tomorrow."
David → all: "Don't forget the meeting at 3."
```

每个 turn 要建模三个维度：

- **Speaker**：谁说的
- **Addressee**：对谁说的（可能多个，可能全员）
- **Content**：说了什么

而且**下一个 speaker 是谁是不确定的**——可能是被 address 的人回应，也可能是第三人插话。

### 1.3 为什么多方更难

LLM 默认是 dyadic 设计的——chat template 里就是 `<|user|>` 和 `<|assistant|>` 两种角色。多方场景要硬塞进这种 template，必须在 content 里手动标 `[Alice]: ...` 这种格式。这种 hack 法有几个问题：

1. **loss 计算混乱**：标准 SFT loss 对所有 token 计算，但你只想让 LLM 学某个 speaker（比如 [Alice]）的发言风格，不学其他 speaker 的。
2. **next-speaker 预测**：标准 LLM 不知道下一个 turn 是谁说的，无法做 addressee 推理。
3. **role-conditioning**：标准 LLM 看不到 role 信息，无法做 persona-consistent 生成。

MuPaS 就是来解决这些问题的。

## 2. "role-masked fine-tuning loss" —— 核心技术

MuPaS 的第一个核心贡献是 **role-masked loss**。具体做法：

```python
def compute_role_masked_loss(input_ids, labels, target_role, all_roles):
    """
    只对 target_role 的发言计算 loss，其他 role 的发言 mask 掉
    """
    role_masks = identify_role_spans(input_ids, all_roles)
    # role_masks[i] = role name of token i (or "non-speaker" for prefix tokens)
    
    loss_mask = torch.zeros_like(labels)
    for i, role in enumerate(role_masks):
        if role == target_role:
            loss_mask[i] = 1
        else:
            loss_mask[i] = 0
    
    # 标准 cross entropy，但只对 mask=1 的 token 算
    losses = F.cross_entropy(logits, labels, reduction='none')
    masked_loss = (losses * loss_mask).sum() / loss_mask.sum()
    return masked_loss
```

### 2.1 为什么 role-masked loss 关键

如果不 mask，LLM 训练时会同时学**所有 role 的语言风格**——这是不对的，因为每个 role 应该有不同的 persona。比如：

- Alice 是技术博士，说话用专业术语
- Bob 是高中生，说话用俚语
- Charlie 是商人，说话夹杂英文

如果 LLM 用一个未 masked 的 loss 学，最后会平均化所有 persona，谁都不像。

Role-masked loss 让 LLM **针对每个 role 学一份独立的 persona**——你想生成 Alice 的发言时，只用 Alice 的 mask 训练；想生成 Bob 的，用 Bob 的 mask。

### 2.2 对你做多角色记忆的应用

你的记忆策略也需要 role-masked——

```python
# 你的多角色 RL 训练
def compute_speaker_aware_rl_loss(trajectory):
    """
    trajectory = [
        (speaker_A_utter, 0),  # 0 = mask（不算梯度，外部输入）
        (model_reasoning, 1),  # 1 = 不 mask
        (model_memory_op, 1),
        (speaker_B_utter, 0),
        ...
    ]
    """
    loss = 0
    for token, mask in trajectory:
        if mask == 1:
            loss += compute_token_grad(token)
    return loss
```

这和 MEM1 的 masked trajectory 是同源思想——**只对模型生成的 token 算 loss/梯度，外部输入 mask 掉**。

但 MuPaS 还多了一层 **role 区分**——同样是模型生成的 token，可能是模型扮演 Alice 生成的，也可能是模型扮演 Bob 生成的，这两个的 loss 应该分别累积、分别用各自 persona 的奖励监督。

## 3. "Two strategies: MuPaS-Speaker, MuPaS-Silence" —— 两种训练范式

MuPaS 提出两种训练策略：

### 3.1 MuPaS-Speaker

只在"当前 role 发言时"训练。其他 role 发言时模型只是"读"，不学。

```python
for turn in dialogue:
    if turn.speaker == target_role:
        # 训练：让模型生成这条
        loss += masked_loss(turn)
    else:
        # 不训练：只更新模型的"看到了这条"的状态
        pass
```

### 3.2 MuPaS-Silence

更激进：在"应该是 target_role 发言但他选择沉默"的时刻也训练。

```python
for turn in dialogue:
    if turn.speaker == target_role:
        loss += masked_loss(turn)  # 学发言内容
    elif this_was_target_role_turn_but_silent(turn):
        loss += silence_loss(turn)  # 学"在这种情况下应该沉默"
```

第二个 loss 是"silence loss"——惩罚模型在不该发言时强行发言，奖励"知道何时闭嘴"。

### 3.3 对你做记忆的扩展启示

**MuPaS-Silence 的核心 idea 是"决定何时不做"**。这对你做记忆操作很有启发——记忆操作的一个 NOOP 动作就是"何时不写记忆"。MuPaS 的训练范式可以借鉴到 NOOP 的学习上：

```python
# 你的扩展
def memory_op_loss(turn):
    if turn.should_write_memory:
        loss += op_loss(turn.predicted_action, turn.gold_action)
    elif turn.should_be_noop:
        loss += noop_loss(turn.predicted_action)  # 惩罚错误的非 NOOP
```

## 4. "MPD 模拟器" —— 训练后的部署

MuPaS 训练完成的模型可以当作 **MPD 模拟器**——给定群组成员、当前对话历史，模拟下一个 turn 应该是谁、说什么。

这种模拟器的价值：

1. **数据增广**：可以生成大量合成 MPD 训练数据。
2. **环境 simulator**：你做 RL 训练时，需要一个对话环境。MPD 模拟器可以做这个 environment。
3. **评测**：next-speaker prediction 的准确率作为多方建模能力的度量。

**对你做 RL 训练的直接价值**：你需要 rollout 环境产生多角色对话训练数据。如果用真实数据有限，可以用 MuPaS-style 模拟器自动生成大量训练对话。这是数据稀缺时的解药。

## 5. "next-speaker prediction" —— 关键辅助任务

MuPaS 的另一个贡献是 next-speaker prediction 作为辅助任务：

```python
# 输入：对话历史
# 输出：下一个 speaker 是谁

next_speaker_loss = cross_entropy(predicted_next_speaker, true_next_speaker)
```

这个任务的意义：

1. **让模型显式建模 speaker 维度**——不只是 implicit 地理解每个 utterance 是谁说的，而是显式预测。
2. **学习 conversational dynamics**——某些情境下 Alice 接话概率高，某些情境 Bob 高。
3. **作为多任务训练的 regularization**——加上 next-speaker prediction 通常能提升 main task 的性能。

**对你做记忆的应用**：

```python
# 多任务训练
total_loss = α * memory_op_loss + β * answer_loss + γ * next_speaker_loss + δ * speaker_attribution_loss
```

其中 `speaker_attribution_loss` 是"判断某条记忆的 speaker 归属是否正确"——你 Idea 1 的核心 reward 的辅助任务版本。

## 6. 实验结果

MuPaS 基于 Qwen2-7B Instruct，主要 metric 是：

- **Response quality**（perplexity、BLEU、ROUGE）
- **Next-speaker accuracy**：相比 Qwen2-7B baseline 提升 10-20%
- **OOD generalization**：换 scenario / topic / role description 仍然 work

具体数字（论文里给的）：

| Metric           | Qwen2-7B | MuPaS-Speaker | MuPaS-Silence |
|------------------|----------|---------------|---------------|
| Response BLEU    | 12.3     | 15.8          | 16.2          |
| Next-Speaker Acc | 42.1     | 58.7          | 61.3          |
| OOD Gen          | 38.5     | 52.4          | 55.1          |

注意 MuPaS-Silence 比 MuPaS-Speaker 都好——证明"学会沉默"是有价值的额外信号。

## 7. Limitation 与反驳

**Limitation 1：纯 SFT，无 RL**。MuPaS 是 supervised fine-tuning，不是 RL。**对你的意义**：你可以用 MuPaS-style SFT 作为 cold-start，然后再做 RL。这是 AgeMem 三阶段课程的第一阶段思路。

**Limitation 2：无记忆**。MuPaS 只做多方生成，不涉及长程记忆。**对你的意义**：你的工作 = MuPaS + 记忆。

**Limitation 3：role 是固定 set**。MuPaS 训练时 role 列表是预定义的，inference 时只能扮演训练时见过的 role。**对你的意义**：你应该考虑 role 的泛化——新 speaker 加入群聊时怎么办。

## 8. 把它们串起来看：MuPaS 的整体设计哲学

MuPaS 的设计哲学是：**多方对话需要 speaker-conditional 建模 → 用 role-masked loss 隔离 persona → 用 next-speaker prediction 显式学 speaker 维度 → 用 silence 学习何时不发言**。

这个哲学的核心是 **"speaker 是一等公民"**——所有训练设计都围绕 speaker 维度。这正是你做多角色记忆时应该坚守的原则。

## 9. 对你的具体借鉴清单

**对所有 Idea（基础设施层面）**：

- **role-masked loss** → 你的 RL training 必备
- **next-speaker prediction** → 你的多任务训练的辅助 head
- **MPD 模拟器** → 你的数据生成/RL 环境

- 三阶段课程的 stage 1 = MuPaS 风格 SFT（学 speaker-aware 基础格式）

## 11. 一个 production-aware 的视角

MuPaS 是一个相对"工程化"的工作——它没有非常 fancy 的方法，但解决了多方 fine-tuning 的一个 fundamental 问题，并提供了 production-ready 的代码。

对你来说，这意味着：

1. **复用其代码**：MuPaS GitHub 上的 role-masked loss 实现可以直接用，省 1-2 周工程时间。
2. **避免重复造轮子**：你不需要从零思考"如何让 LLM 学 speaker 维度"，MuPaS 已经给了答案。
3. **聚焦你的真正创新**：把精力放在"memory + speaker"的结合上，把"speaker 维度本身的建模"借鉴自 MuPaS。

这种"借用基础工作 + 创新结合点"是 ML 论文的标准模式。AAAI 审稿人会欣赏这种 thoughtful integration。
