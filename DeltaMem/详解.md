# DeltaMem 详解

> 论文：DeltaMem: Towards Agentic Memory Management via Reinforcement Learning
> arxiv：2604.01560，2026 年 4 月
> 在本工作中的角色：**SpeakerMem-R1 v2 的核心借鉴对象——dense memory state reward 范式**

---

## 1. 它在解决什么问题

### 1.1 一句话

DeltaMem 把"记忆管理"形式化为 **persona-centric 端到端任务**，并提出一个全新的 **dense process reward**——**Memory-based Levenshtein Distance**——彻底绕开 Memory-R1 风格的"下游 QA 答对才有 reward"的稀疏奖励陷阱。

### 1.2 它批判了谁

DeltaMem 的核心 motivation 是直接打 Memory-R1：

> Memory-R1 之类的方法用下游任务表现作为 reward signal 训练 memory manager 用 RL，但这会引入难以承受的稀疏奖励问题。

也就是说，原本你设想"memory 写错了 → 答错了 → 没 reward"是一个数 step 才能传一次的极弱信号，对 GRPO 这种 group-relative 算法尤其致命（同 group 内可能 G 条都是 0 reward，gradient 直接消失）。

DeltaMem 的解法：**别等 QA 答对，写 memory 的那一刻就给 reward**——奖励信号密集到每个 memory operation 之后都能计算。

### 1.3 它对我们意味着什么

我们的 SpeakerMem-R1 v1 原版用的奖励是"R_task + R_attr + R_compr + R_aud + R_RIF"，其中 R_task 是 outcome（稀疏），R_attr 等是过程但都是 hand-crafted heuristic。**DeltaMem 给出了一个更优雅、更普适的过程奖励范式**——用 ground-truth memory state 与 predicted state 的距离做 reward。这意味着我们 v2 应该把 R_state（speaker-aware Levenshtein）作为**主导信号**（DeltaMem 经验：权重 0.8）。

## 2. 技术摘要：Memory-based Levenshtein Distance 是什么

### 2.1 经典 Levenshtein 距离回顾

经典 Levenshtein 距离是 edit distance——计算把字符串 A 变成字符串 B 需要多少次"插入/删除/替换"操作。例如 `kitten → sitting` 是 3 步。

但 memory state 不是字符串——它是一组结构化 fact entries（如 "Alice likes coffee"、"Bob works at OpenAI"）。如何把 Levenshtein 推广到 fact set？

### 2.2 DeltaMem 的推广

DeltaMem 把 memory state 看成 **fact set**，然后定义 fact-level 的 edit distance：

**Step 1: 计算 Δ（增量）**

```
prev_state = memory state at turn t-1
target_state = ground truth memory state at turn t
pred_state = our model's predicted memory state at turn t

Δ_target = target_state - prev_state   # 这一 turn 应该新增/修改的 facts
Δ_pred = pred_state - prev_state       # 这一 turn 我们模型实际新增/修改的 facts
```

> 关键观察：奖励信号不是 absolute memory state，而是 **delta**。这是 DeltaMem 名字的由来。

**Step 2: 算 Δ_pred 和 Δ_target 之间的 similarity matrix**

对 Δ_pred 中每条 fact f_p 和 Δ_target 中每条 fact f_t，用 sentence embedding 算 cosine similarity：

```
M[i,j] = cos_sim(embed(Δ_pred[i]), embed(Δ_target[j]))
```

**Step 3: Optimal transport 匹配**

把 M 当成一个 bipartite 匹配问题，用 Hungarian algorithm 或 Sinkhorn 找最优 one-to-one 匹配：

```
matching = optimal_transport(M)
```

得到 pairs of (i, j) such that 总相似度最大化。

**Step 4: 阈值过滤**

```
matched_pairs = [(i,j) for (i,j) in matching if M[i,j] >= τ]
```

低于 τ 的对算"未匹配"——不算 true positive。

**Step 5: 局部 lexical fidelity**

光靠 embedding similarity 不够——可能 "Alice likes coffee" 和 "Bob likes coffee" embedding 相似但 entity 错了。所以加 lexical check：

```
lex_score(f_p, f_t) = (keywords of f_t) ∩ (tokens of f_p) / |keywords of f_t|
```

然后把 lex_score 乘进 similarity matrix：

```
M_final[i,j] = M[i,j] * lex_score(Δ_pred[i], Δ_target[j])
```

**Step 6: Soft-precision / soft-recall / F1**

```
soft_precision = (sum of matched similarities) / |Δ_pred|
soft_recall = (sum of matched similarities) / |Δ_target|
F1 = 2 * P * R / (P + R)
```

这个 F1 就是 R_state_transition。

### 2.3 完整奖励公式

DeltaMem 的最终训练奖励：

```
R_total = 0.1 * R_format + 0.1 * R_retrieval + 0.8 * R_state_transition
```

其中：
- R_format：output JSON 格式合法（0/1）
- R_retrieval：retrieve 出的 fragment 命中 ground truth（F1）
- R_state_transition：上面那个 Levenshtein F1（**主导信号**）

权重 0.8 给 state transition——这是 DeltaMem 的核心 design choice，证明 **dense 状态对齐信号比 outcome QA 信号更适合训 memory manager**。

## 3. 训练框架

### 3.1 RL 算法

DeltaMem 用标准 GRPO。重点在 reward 设计，不在 RL 算法本身。

### 3.2 训练数据

合成数据 —— 用 LLM 生成多 turn 对话 + ground truth memory state at each turn。这是 DeltaMem 自己承认的局限：可能有 LLM bias。

### 3.3 评测

三个 benchmark：

| Benchmark   | 类型              | DeltaMem-8B-RL 表现 |
|-------------|-------------------|---------------------|
| LoCoMo      | 长程多 session 对话 | 显著超 Memory-R1    |
| PersonaMem  | 动态偏好跟踪      | 显著超 Memory-R1    |
| HaluMem     | 操作级 extraction/update/QA | 显著超 Memory-R1 |

具体数字论文里有，本详解不复述（重要的是范式）。

## 4. 对 SpeakerMem-R1 v2 的具体借鉴

### 4.1 直接借鉴：Speaker-aware Levenshtein

DeltaMem 是 dyadic 的——只有 user-assistant 两方。多方场景下，memory state 应该按 **speaker bucket**：

```python
def speaker_aware_levenshtein_reward(pred_state, target_state, speakers):
    """
    pred_state, target_state: dict[speaker_id, list of facts]
    speakers: list of speaker_ids
    """
    per_speaker_scores = []
    for s in speakers:
        f1 = deltamem_levenshtein_f1(pred_state[s], target_state[s])
        per_speaker_scores.append(f1)
    
    # mean over speakers
    base = mean(per_speaker_scores)
    
    # 关键：min-aggregator 惩罚"忽略某个 speaker"
    # 否则模型可能集中精力把主 speaker 写对、忽略外围
    worst_speaker_penalty = 0.5 * min(per_speaker_scores)
    
    return base + worst_speaker_penalty
```

这是 SpeakerMem-R1 v2 的 **R_state 核心组件**。

### 4.2 间接借鉴：state-diff 范式

DeltaMem 教给我们一个深刻原则：**记忆 RL 的奖励应该比较"应该发生的状态变化"和"实际发生的状态变化"，而不是比较"最终任务输出"**。

这个原则可以推广到我们 v2 的其他维度：

- **R_attribution** 也可以做 delta-form：比较 ground truth 中"这一 turn 新归属给 Alice 的 facts" vs 模型实际归属给 Alice 的 facts
- **R_audience** 也可以 delta-form：比较"这一 turn 输出应该被多少 speakers 看到" vs 模型实际可见性设置

也就是说，**整个 v2 奖励体系都可以 delta-化**，奖励信号会大幅增密。

### 4.3 跨 speaker 泄漏惩罚

DeltaMem 是 dyadic，没有"泄漏"概念。但多方场景下，**模型可能把 Alice 的 fact 错误归属到 Bob 的 memory**。我们要加一项：

```python
def cross_speaker_leakage_penalty(pred_state, target_state):
    """计算 pred[s] 中混入的本应属于其他 speaker 的 facts"""
    leak = 0
    for s in speakers:
        for f in pred_state[s]:
            # if f 在 target 中属于其他 speaker，扣分
            for s_other in speakers:
                if s_other != s and f in target_state[s_other]:
                    leak += 1
    return -leak / total_facts
```

### 4.4 阈值 τ 的多方扩展

DeltaMem 的 τ 是 global threshold。多方场景下，不同 speaker 的"alignment 难度"不同（高频 speaker vs 边缘 speaker）。可以做 **per-speaker 自适应 τ**：

```
τ_s = base_τ + adjustment_based_on_speaker_freq(s)
```

这是 v2 的一个 ablation 维度。

## 5. DeltaMem 的局限（我们的差异化机会）

### 5.1 局限 1：纯 dyadic

只有 user-assistant 两方。完全不处理：
- speaker attribution
- audience adaptation
- 多 speaker 间的隐私边界

**→ 我们的 multi-party 设定就是 vs DeltaMem 的核心差异化**

### 5.2 局限 2：纯合成数据

DeltaMem 论文自己承认训练数据全是 LLM 合成的，可能有 social bias。

**→ 我们可以用 SHARE pipeline 抽取的电影对话（半真实数据）作为补充**

### 5.3 局限 3：没考虑遗忘

DeltaMem 的 state 是单调增长的，没有 active 遗忘动作。但长程对话中"该忘的得忘"。

**→ 我们结合 MOOM 的竞争-抑制机制 + DeltaMem 的 reward 范式，做 forgetting-aware Levenshtein**

具体：在 R_state 中加入 forgetting term：

```
R_forget = - leftover_obsolete_facts_score
         = how many "已过期 facts" 还留在 memory
```

### 5.4 局限 4：没和 outcome reward 联合

DeltaMem 权重 0.8 给 state，但 outcome reward 只有 0.0（没有！）——实际上它把 retrieval 0.1 当成弱 outcome proxy。我们 v2 应该 **同时保留 outcome reward (R_task) 和 state reward (R_state)**：

```
R = 0.5 * R_state + 0.3 * R_task + 0.2 * R_other
```

这样既有"写对 memory"的过程信号，也有"答对问题"的最终对齐。

## 6. 伪代码：v2 R_state 完整实现

```python
import torch
from scipy.optimize import linear_sum_assignment
from sentence_transformers import SentenceTransformer

embedder = SentenceTransformer('all-mpnet-base-v2')

def fact_to_embedding(fact: str):
    return embedder.encode(fact)

def levenshtein_state_f1(delta_pred, delta_target, tau=0.6):
    """DeltaMem-style F1 for one speaker"""
    if not delta_pred and not delta_target:
        return 1.0
    if not delta_pred or not delta_target:
        return 0.0
    
    P = torch.tensor([fact_to_embedding(f) for f in delta_pred])
    T = torch.tensor([fact_to_embedding(f) for f in delta_target])
    
    # cosine similarity matrix
    P_norm = P / P.norm(dim=1, keepdim=True)
    T_norm = T / T.norm(dim=1, keepdim=True)
    M = (P_norm @ T_norm.T).numpy()
    
    # lexical fidelity
    for i, fp in enumerate(delta_pred):
        for j, ft in enumerate(delta_target):
            keywords = extract_keywords(ft)
            lex = sum(1 for k in keywords if k in fp) / max(len(keywords), 1)
            M[i, j] *= lex
    
    # optimal transport (Hungarian: negate for maximization)
    row_ind, col_ind = linear_sum_assignment(-M)
    
    matched_sims = [M[r, c] for r, c in zip(row_ind, col_ind) if M[r, c] >= tau]
    
    soft_prec = sum(matched_sims) / len(delta_pred)
    soft_rec = sum(matched_sims) / len(delta_target)
    
    if soft_prec + soft_rec == 0:
        return 0.0
    return 2 * soft_prec * soft_rec / (soft_prec + soft_rec)


def speaker_aware_state_reward(
    pred_memory: dict,    # speaker_id -> list of facts
    target_memory: dict,
    prev_memory: dict,
    speakers: list,
    tau: float = 0.6,
    min_speaker_weight: float = 0.5,
):
    """Speaker-aware Levenshtein reward for SpeakerMem-R1 v2"""
    per_speaker = []
    for s in speakers:
        delta_pred = set(pred_memory.get(s, [])) - set(prev_memory.get(s, []))
        delta_target = set(target_memory.get(s, [])) - set(prev_memory.get(s, []))
        f1 = levenshtein_state_f1(list(delta_pred), list(delta_target), tau)
        per_speaker.append(f1)
    
    if not per_speaker:
        return 0.0
    
    base = sum(per_speaker) / len(per_speaker)
    worst = min(per_speaker)
    
    # 关键: worst-speaker penalty 防止模型忽略某 speaker
    return base + min_speaker_weight * worst


def cross_speaker_leakage(pred_memory, target_memory, speakers):
    leak = 0
    total = sum(len(target_memory.get(s, [])) for s in speakers)
    for s in speakers:
        for f in pred_memory.get(s, []):
            for s_other in speakers:
                if s_other != s and f in target_memory.get(s_other, []):
                    leak += 1
    return -leak / max(total, 1)


def speakermem_r1_v2_reward(traj, ground_truth):
    speakers = ground_truth['speakers']
    
    R_state = speaker_aware_state_reward(
        pred_memory=traj.final_memory,
        target_memory=ground_truth['memory'],
        prev_memory=traj.initial_memory,
        speakers=speakers,
    )
    R_leak = cross_speaker_leakage(traj.final_memory, ground_truth['memory'], speakers)
    R_task = llm_judge(traj.final_answer, ground_truth['answer'])
    R_attr = speaker_attribution_f1(traj.attributions, ground_truth['attributions'])
    R_aud = audience_adaptation_score(traj.outputs, ground_truth['audience'])
    R_compr = - len(traj.final_memory_serialized) / max_tokens
    
    return (0.5 * R_state +     # 主导：DeltaMem 升级
            0.2 * R_leak +      # 多方独有
            0.15 * R_task +     # outcome
            0.1 * R_attr +
            0.05 * R_aud +
            0.0 * R_compr)
```

## 7. 对比表：v1 vs v2 中的 reward 设计

| 维度 | v1 reward | v2 reward (借鉴 DeltaMem) |
|------|-----------|---------------------------|
| 主导信号 | R_task (outcome, 稀疏) | **R_state (process, dense)** |
| 信号密度 | 1 次/对话 | **N 次/对话**（每 turn 一次） |
| 收敛性 | 担心稀疏 | DeltaMem 已验证可收敛 |
| 计算 | LLM judge 慢 | embedding + Hungarian 快 |
| 训练成本 | 高（每 group 都要 LLM-judge） | **低**（embedding 缓存可复用） |

## 8. Related Work 写法建议

在我们 paper 的 Related Work 中，DeltaMem 的位置：

> Recent work has begun addressing the sparse reward problem in memory RL. DeltaMem [cite] proposes a Memory-based Levenshtein Distance reward that compares predicted and ground-truth memory state changes via optimal transport over embedded fact sets, demonstrating dense process reward outperforms outcome-only training in dyadic settings (LoCoMo / PersonaMem / HaluMem). However, DeltaMem assumes a single-user persona-centric setup; its state-diff formulation does not account for speaker grounding, audience adaptation, or cross-speaker information leakage—all of which are central to multi-party group memory. Our work extends DeltaMem's dense state reward to a **speaker-aware bucketed Levenshtein** formulation and introduces a cross-speaker leakage penalty, yielding the first end-to-end RL memory training framework operating on multi-party benchmarks (GroupMemBench / EverMemBench / SocialMemBench).

## 9. 整体设计哲学

DeltaMem 的核心洞察："Don't reward the answer, reward the memory."

我们 v2 的延展：**"Don't reward just the memory, reward per-speaker memory and penalize speaker leakage."**

这个递进关系是清晰的：
- Memory-R1: reward the answer (outcome only)
- DeltaMem: reward the memory state (process, dense, dyadic)
- **SpeakerMem-R1 v2: reward per-speaker memory state + penalize cross-speaker leakage (multi-party, speaker-grounded)**
