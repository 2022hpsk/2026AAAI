## Part 1: 需要补充调研的内容

按"对论文成败的关键度"排序。这些是深度调研报告里未充分展开、但你做 AAAI 论文时必须明确的细节。

### 1.1 [高优] DeltaMem 论文的具体内容

**为什么重要**：DeltaMem 是直接批判 RL 记忆"稀疏奖励难收敛"的工作，它的具体论证强度决定你 Idea 1/3 的 motivation 强度。

**需要调研的具体问题**：

- DeltaMem 的实验设计是怎样的？给出了什么具体收敛曲线/失败案例？
- DeltaMem 自己提的 mitigation 方案是什么？过程奖励、demonstration learning 还是其他？
- DeltaMem 在哪些 benchmark 上验证？和 Memory-R1/AgeMem 数字对比？
- DeltaMem 作者是否就是 AgeMem/Memory-R1 同一群人在自我批评？还是不同团队挑战？

**调研方法**：直接读 arXiv 2604.01560 全文 + 看 GitHub 仓库（如果有）

### 1.2 [高优] step-wise GRPO 的具体技术细节

**为什么重要**：你做 Idea 1 必用 step-wise GRPO，但 AgeMem 论文里这一部分写得不够细。

**需要调研的具体问题**：

- step-wise 的 "step" 边界怎么定义？按 turn、按动作、按 sub-trajectory？
- group sampling 在 step-wise 下怎么做？sample G 条 sub-trajectory 还是 G 条完整 trajectory 然后切？
- advantage 估计的具体公式？token-level vs step-level？
- 与 standard GRPO 的具体 code-level diff？

**调研方法**：
- 读 AgeMem 论文实验细节
- 找 AgeMem GitHub 代码（如果开源）
- 读 trl/verl 库的 GRPO 实现 + step-wise 扩展

### 1.3 [高优] GroupMemBench / EverMemBench / SocialMemBench 的具体数据 schema

**为什么重要**：你必须在这三个 benchmark 上跑实验，但深度调研报告里没给详细数据格式。需要明确：

- 每条数据的具体 JSON schema 是什么样？
- 评测代码用什么 metric？exact match、F1、还是 LLM-judge？
- 是否需要 GPU/TPU 跑评测？时间预算？
- 数据是否完整开源还是部分？

**调研方法**：
- 访问每个 benchmark 的 GitHub（论文里有链接）

## Part 2: 有价值的 AAAI Idea

在原深度调研三 idea（SpeakerMem-R1 / GroupPolicy / GroupReward）基础上，结合 18 个详解给出 **细化版 + 新增 idea**。

### 2.1 [推荐主推] SpeakerMem-R1（细化版）

原 idea 的细化补强（来自详解的具体借鉴）：

**核心方法增强**：

1. **三阶段课程（from AgeMem + MuPaS）**：
   - Stage 1: MuPaS-style role-masked SFT（学 speaker token + persona 一致）
   - Stage 2: AgeMem 风格 LTM 单独 RL（学 speaker-grounded memory writing）
   - Stage 3: 完整 RL with memory ops（multi-context GRPO）

2. **混合记忆架构（from Mem-α + MemAgent）**：
   - speaker private core / episodic / semantic（Mem-α 分层）
   - 每 speaker 一份 MemAgent-style growing memory（overwrite）
   - 群组 shared insight（G-Memory 的 insight graph 借鉴）

3. **奖励设计（from AgeMem + Mem-α + DeltaMem）**：
   ```
   R = R_task + α·R_attr + β·R_compression + γ·R_audience + δ·R_RIF
   ```
   - R_task: 下游 QA
   - R_attr: speaker 归属正确性（针对 GroupMemBench user-implicit）
   - R_compression: 记忆压缩度（from Mem-α β=0.05）
   - R_audience: 受众适配度（针对 GroupMemBench audience-adapted）
   - R_RIF: 竞争-抑制遗忘 reward shaping（from MOOM）

4. **训练算法（from MemSearcher + AgeMem + WebAgent-R1 + RAGEN）**：
   - 主算法：step-wise GRPO（AgeMem 风格 credit assignment）
   - 扩展：multi-context GRPO（MemSearcher 风格 evolving context）
   - 工程：异步 rollout + 动态压缩（WebAgent-R1）
   - 防 Echo Trap：KL 正则 + diversity reward（RAGEN）

5. **评测全谱（from 三 benchmark）**：
   - 主: GroupMemBench（六类）、EverMemBench（三维）、SocialMemBench（五失败模式）
   - dyadic 对照: LOCOMO + LongMemEval
   - 隐式个性化: PersonaMem-v2

**优势**：每个组件都有清晰的祖先 paper，方法论扎实。

**风险**：组件众多可能让审稿人质疑 "incremental"。

**应对**：用一个 unified theoretical frame 把组件串起来——比如 "speaker-grounded learnable structured memory" 作为统一概念。

### 2.2 [推荐 secondary] GroupPolicy（细化版）

原 idea 的细化补强：

**核心方法增强**：

1. **架构（from Collaborative Memory + G-Memory）**：
   - Collaborative Memory 的双层（private/shared）+ provenance
   - G-Memory 的三层（interaction/query/insight）
   - 合并为 5 层结构：speaker_private_episodic / speaker_private_core / group_interaction / group_query / group_insight

2. **动作空间**：
   ```python
   actions = [
       WRITE_PRIVATE(content, speaker_id, layer),
       WRITE_SHARED(content, audience_set, layer),
       PROMOTE(fragment_id, from_layer, to_layer),  # 重要：跨层 promotion
       DEMOTE(fragment_id, ...),  # 反向
       READ_CROSS(asker_id, target_speaker_id),  # 受 access control 约束
   ]
   ```

3. **奖励（明确隐私维度）**：
   ```
   R = R_collab + μ·R_efficiency - η·R_leak - ν·R_isolation
   ```
   - R_collab: 协作任务成功
   - R_leak: 隐私泄露（用 SocialMemBench failure mode 4 类）
   - R_isolation: 过度私有化（信息该共享但没共享，导致协作失败）

4. **训练**：M-GRPO（WebAgent-R1）+ multi-context（MemSearcher）混合

**优势**：解决一个真实 production 痛点（隐私），创新性高。

**风险**：隐私 metric 怎么定义需要精心设计。

**应对**：参考差分隐私 / federated learning 文献，借用 leak metric 定义。

### 2.3 [推荐备选] GroupReward（细化版）

原 idea 的细化：

**核心增强**：

1. **PRM 设计（结合各 benchmark 维度）**：
   - 6 维 PRM head，对应 GroupMemBench 六类 query：
     - multi-hop reasoning
     - knowledge update
     - term ambiguity
     - user-implicit reasoning
     - temporal reasoning
     - abstention
   - 每个 head 独立训练，inference 时 weighted combine

2. **PRM 训练数据（from 多源）**：
   - SHARE pipeline 抽取的电影对话
   - GroupMemBench / EverMemBench 自带数据
   - 自己合成的多角色对话（用 MuPaS-style simulator）

3. **PRM 标注（半自动）**：
   - LLM judge 自动标
   - 5% 人工抽审 high-impact case
   - LLM judge 用 GroupMemBench 失败模式 prompt（more focused）

4. **数据贡献**：
   - 1 万 + 训练对话
   - 每对话带操作级 PRM 标注
   - 开源 + 附 PRM checkpoint

**优势**：双贡献（方法+数据）抗审稿风险。

**风险**：工作量最大，可能 12+ 周。

**应对**：PRM 可以小（3B），数据可以分批发布（先 method paper + benchmark paper 拆两篇）。

### 2.4 [新增] Idea 4: SpeakerForget-RL —— RL 学习的 speaker-aware 遗忘

**核心创新**：把 MOOM 的"竞争-抑制遗忘"做成 **speaker-aware learnable forgetting policy**。

**动机**：
- MOOM 的竞争-抑制是 hardcoded（固定阈值、固定衰减率）
- 多角色场景下，**不同 speaker 的记忆应该有不同遗忘速率**——重要 speaker 的记忆保留更久、外围 speaker 的可以更快忘
- 没人做过 RL learnable speaker-aware forgetting

**方法**：

```python
forget_action = SUPPRESS(fragment_id, suppression_strength)

# 每个 fragment 有 activation 分数
# RL 学：什么时候 suppress 哪条 fragment
# 奖励：长期 QA 准确率（不能因为忘了关键信息掉点）
```

**与 MOOM 的 delta**：
- MOOM: fixed forgetting threshold across all fragments
- 你的: learned, per-speaker, contextual forgetting

**benchmark**：MOOM 的 ZH-4O（中文角色扮演）+ EverMemBench（长程多群组）

**优势**：narrow 但 deep 的技术贡献，容易做出明确 delta。

**风险**：可能 too narrow——只解决 forgetting 这一面。

**应对**：和 SpeakerMem-R1 合并——作为一个 ablation 维度。

### 2.5 [新增] Idea 5: MultiPartyPRM —— 多方对话的过程奖励模型

**核心创新**：不做 RL 训练，只做 **PRM benchmark + RPM**。

**动机**：
- 当前 RL 记忆方法都用 outcome reward
- 没人系统性研究 process reward 在多方场景的设计
- 一个好的 PRM 可以 benefit 整个领域

**方法**：
1. 构造 multi-party PRM training data（用 GroupMemBench style data + 操作级标注）
2. 训练 7B PRM
3. 评测 PRM 自身的质量
4. demonstrate PRM 能让 outcome-only RL 收敛更快

**与 Idea 3 的差异**：
- Idea 3 是 method paper（用 PRM 训 policy）
- Idea 5 是 PRM-focused paper（PRM 自身设计 + 评测）

**适合投递**：NeurIPS Datasets & Benchmarks track（PRM-only paper）

### 2.6 [新增] Idea 6: SpeakerProfile-Mem —— 隐式 speaker profile 学习

**核心创新**：把 PersonaMem-v2 的"隐式偏好"扩展到多 speaker 场景。

**动机**：
- PersonaMem-v2 证明 dyadic implicit personalization 难（37-48%）
- 多 speaker 场景下，模型要 simultaneously 学 N 个 implicit profile，更难
- 没人做过

**方法**：
1. 每 speaker 一份 growing profile（PersonaMem-v2 启发）
2. RL 学 profile 更新 policy
3. 评测：每个 speaker profile 的隐式 inference 准确率

**benchmark**：构造 multi-party PersonaMem-v2

**优势**：bridge personalization 和 multi-party 两个领域，新角度。

**风险**：多 speaker 隐式偏好 ground truth 怎么定义？

**应对**：用 SHARE / 合成数据，每个 character 设计明确 implicit traits。

## Part 3: 方法学合成 + 风险地图

### 3.1 方法学合成 —— 一个统一的 "stack"

把 18 个详解的技术合并成你方法的 **technical stack**：

```
┌─────────────────────────────────────────────────────────┐
│ APPLICATION: Multi-party group chat memory management   │
├─────────────────────────────────────────────────────────┤
│ EVALUATION:                                             │
│   - Main: GroupMemBench + EverMemBench + SocialMemBench │
│   - Dyadic compat: LOCOMO + LongMemEval                 │
│   - Personalization: PersonaMem-v2                      │
├─────────────────────────────────────────────────────────┤
│ ARCHITECTURE:                                           │
│   Layer 5: Group insight (G-Memory inspired)            │
│   Layer 4: Group shared episodic                        │
│   Layer 3: Per-speaker shared profile (PersonaMem-v2)   │
│   Layer 2: Per-speaker private episodic                 │
│   Layer 1: Per-speaker private core                     │
│                                                         │
│   Per layer:                                             │
│   - Mem-α core/episodic/semantic structure              │
│   - SA-LLM special speaker tokens                       │
│   - Collaborative Memory provenance attributes          │
│   - MemAgent overwrite (for free-text memory)           │
│   - Mem0/A-MEM CRUD ops (for structured memory)         │
├─────────────────────────────────────────────────────────┤
│ ACTIONS:                                                │
│   {ADD, UPDATE, DELETE, NOOP, SUMMARY, FILTER,          │
│    PROMOTE, DEMOTE, SUPPRESS, READ_CROSS}               │
│   × {speaker_id, audience, layer}                       │
├─────────────────────────────────────────────────────────┤
│ REWARDS:                                                │
│   - Outcome: R_task (LLM-as-judge)                      │
│   - Process: R_attribution (speaker grounding)          │
│   - Process: R_audience (audience adaptation)           │
│   - Process: R_compression (Mem-α β=0.05)               │
│   - Process: R_RIF (MOOM-inspired forgetting)           │
│   - Privacy: R_leak (if Idea 2)                         │
├─────────────────────────────────────────────────────────┤
│ TRAINING:                                               │
│   Stage 1: MuPaS-style role-masked SFT                  │
│   Stage 2: LTM-only RL (AgeMem stage 1)                 │
│   Stage 3: Full end-to-end RL                           │
│                                                         │
│   Algorithm: step-wise GRPO (AgeMem)                    │
│             + multi-context GRPO (MemSearcher)          │
│             + M-GRPO (WebAgent-R1)                      │
│             + masked trajectory (MEM1)                  │
│   Mitigations:                                          │
│     - KL regularization (vs RAGEN Echo Trap)            │
│     - Async rollout (WebAgent-R1)                       │
│     - PRM augmentation (vs DeltaMem critique)           │
├─────────────────────────────────────────────────────────┤
│ DATA:                                                   │
│   - Training: synthesized via MuPaS simulator + SHARE   │
│   - Validation: GroupMemBench dev set                   │
│   - Test: All three multi-party benchmarks              │
└─────────────────────────────────────────────────────────┘
```

这个 stack 不是 "我都用"，而是 **可选 component library** —— 实际方法会从中挑 8-10 个 component 组合。

### 3.2 风险地图

按风险级别排序：

#### 风险 1：[高] AAAI-27 timing 紧

- **2026 年 7 月 28 日交全文**
- 你现在（2026 年 6 月）只剩 ~2 个月
- baseline 复现 + 主方法训练 + 消融 + 写作

**应对**：
- 立即开始：复现 AgeMem / Memory-R1 / Mem0
- 6-7 月：方法迭代 + 主实验
- 7 月底冲刺写作
- **如果赶不上 7 月 28 日，转 EMNLP 2026 备份**

#### 风险 2：[高] GroupMemBench/EverMemBench 作者团队的 follow-up

- 他们大概率出 method paper on their own benchmark
- 如果先于你 publish，你的优先级被压

**应对**：
- 尽早 arXiv 占位（即使方法不完美）
- 强调 RL angle（他们可能做 retrieval）
- 建立合作 vs 竞争（联系作者）

#### 风险 3：[中] DeltaMem 揭示的稀疏奖励问题

- 你的方法可能复现这个问题
- 训练曲线不收敛 → 主实验失败

**应对**：
- 用 PRM 过程奖励（Idea 3 思路融入）
- 用三阶段课程缓解
- 用 demonstration warm-start
